# 결제 시스템 포트폴리오 로드맵

> 결제/페이먼츠 도메인 신입~주니어 백엔드 채용을 목표로,
> **실제 채용공고 15건 + 기업 기술블로그/컨퍼런스 발표 + 공식 PG 문서**를 분석해 만든 로드맵 문서 세트.
>
> 조사 기준일: 2026-07-05

## 이 문서 세트의 결론 (TL;DR)

**"PG 연동해서 결제 성공"까지는 흔한 프로젝트다. (아임포트/포트원 연동 블로그 글만 수십 건)**

채용하는 쪽이 군침 흘리는 포트폴리오는 아래 5개 축에서 갈리고, 이 5개 축은 결제 도메인 면접 단골 질문과 정확히 일치한다:

| # | 차별화 축 | 대응하는 공고 문구 | 대응하는 면접 질문 |
|---|---|---|---|
| 1 | **실패 설계** — 멱등키 + 상태머신 + 보상 트랜잭션(망취소) | 카카오페이 "장애 복구 시나리오 구축", 토스 "장애 해결 경험"(필수) | "PG 승인 후 DB 롤백되면?", "타임아웃 시 중복 환불 어떻게 막나?" |
| 2 | **보안 검증** — 웹훅 검증, 금액 위변조 검증 | 카카오페이 "Secure Coding 생활화"(우대) | "클라이언트가 금액을 조작하면?" |
| 3 | **정산 + 대사** — Spring Batch 정산, PG 대사 배치, 복식부기 원장 | 토스페이먼츠 "오차 없이 계산하는 정산 시스템", 크림 "대사 시스템 개발" | "내부 기록과 PG 기록이 다르면?" |
| 4 | **수치가 있는 성능 개선** — 부하테스트 리포트 (목표 TPS → 병목 계측 → 개선 → 전후 수치) | 토스페이먼츠 "수십배 트래픽에도 무너지지 않는", 카카오페이 "병목 진단"(우대) | "동시에 1000명이 결제하면?" |
| 5 | **문서화** — ADR, 트러블슈팅, 장애 재현 실험 리포트 | 토스페이먼츠 이력서 가이드 "문제 해결을 최대한 구체적으로" | (이력서/포트폴리오 서류 통과 자체) |

## 프로젝트 기술 스택 (공고 분석 기반 확정)

조사한 공고 15건 중 사실상 전부(13~14건)가 **Java/Kotlin + Spring Boot** 였다. 결제 도메인 = JVM/Spring 생태계가 표준.

- **언어/프레임워크**: Java 21 (또는 Kotlin) + Spring Boot 3.x — 공고 표준
- **DB**: MySQL + JPA(도메인모델 방식 — 다날/우아한형제들 공고의 공통 문구) + QueryDSL
- **인프라**: Redis(캐시·분산락), Kafka(이벤트), Spring Batch(정산)
- **PG**: 토스페이먼츠 (테스트 키 무료, 문서 품질 국내 최고, 채용 타깃사 그 자체)
- **테스트/성능**: JUnit5 + Testcontainers, k6(부하테스트), Resilience4j(서킷브레이커)
- **관측성**: Prometheus + Grafana, 구조화 로깅(traceId)
