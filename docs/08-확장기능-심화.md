# 08. 확장 기능 — 현직자가 "오, 이걸 했어?"라고 할 심화 아이템 10종

> 기본기(Phase 0~6)를 끝낸 뒤 고를 확장 기능.
> 공통 원칙: **어필이 극대화되는 지점은 전부 "PG가 책임지지 않는 영역"** — 실패 처리, 상태 관리, 대사, 보상, 금액 배분, 규제 반영.
> dunning·환불 우선순위·멀티 PG failover는 국내 공개 자료 자체가 드물어, 직접 설계·문서화하면 희소성이 가장 크다.

## 1. 멀티 PG 스마트 라우팅 + 자동 failover ⭐ 최고 희소성

### 왜 만드나
- 국내 상위 PG사에서 실제 약 1시간 API 장애 사례 존재 — 단일 PG 연동이면 그 시간 동안 매출 손실 직격
- 멀티 PG failover를 직접 설계한 공개 포트폴리오는 극히 드묾 (조사 10개 주제 중 가장 희소)
- Phase 2에서 만든 서킷브레이커 위에 자연스럽게 얹히는 확장

### 실무 구조
- **포트원 스마트 라우팅 방식**: 채널 그룹 생성 → `channelGroupId` 발급 → 결제 시 PG 대신 그룹 ID 전달 → **그룹 내 채널 비율에 따라 가중 랜덤 배분**
- **슈퍼빌링키**: 빌링키 발급 시 그룹 내 모든 PG에 발급 시도 → 하나의 통합 빌링키에 매핑 → **정기결제도 PG 전환/분산 가능**
- 포트원의 failover도 사실은 "콘솔 **수동** 전환이 매우 빠른 것"(자체 구축 1시간 → 10초) — **완전 자동 failover를 구현하면 포트원보다 한 발 더 나간 설계**
- **장애 감지**: 롤링 60초 윈도우로 PG별 에러율·응답시간 집계 → 5xx/타임아웃 비율 5~10% 초과 시 전환. 서킷브레이커와 결합 (OPEN → 대체 PG, HALF_OPEN → 회복 판정)
- **수수료 최적화 라우팅**: 결제수단 × PG 수수료 매트릭스로 최저비용 채널 선택. 일반 카드 3.2~3.5%, 영세·중소 우대 0.5~1.5% (매년 1·7월 갱신)
- 해외 비교: Stripe Vault and Forward API(자사 토큰화 후 타 PSP 포워딩), Adyen(로컬 매입 강점)

### 최중요 실무 포인트: 재시도 vs 전환 분류 매핑 테이블

| 거절 유형 | 예시 | 대응 |
|---|---|---|
| **Hard decline** | 도난/분실/무효 카드 | 어느 PG로 가도 실패 — **재시도·전환 모두 금지, 즉시 실패** |
| **Soft decline** | 잔액부족, 한도초과, 일시 오류 | 재시도 유효 — 워터폴 라우팅으로 15~22% 회수 |
| **Processor error** | PG 자체 5xx, 게이트웨이 오류 | **즉시 타 PG 전환** |
| **타임아웃** | 응답 없음 | 실패 아님(Unknown) — **멱등키 + 원거래 상태 조회 후** 재시도. 확인 없이 재시도하면 이중결제 |

### 구현 체크리스트
- [ ] PG 어댑터 인터페이스 추상화 (기존 토스페이먼츠 연동을 어댑터 1호로 리팩토링)
- [ ] **mock PG 2~3개 직접 제작** — 지연·에러율을 API로 주입 가능하게 (외부 의존 없이 완주하는 열쇠)
- [ ] 거절 코드 분류 테이블 (hard/soft/processor/timeout) + 대응 정책
- [ ] 60초 롤링 윈도우 헬스 집계 + 서킷 연동 자동 전환
- [ ] 가중치 기반 배분 (수수료 최적화 모드)
- [ ] **킬러 데모**: mock PG 하나를 죽였을 때 자동 전환되는 장애 주입 시연 — 전환 소요 시간, 실패 건수 0 증명 리포트

### 평가
- **난이도**: 중 — mock PG 구성 시 3~4주. "타임아웃 시 상태 미확정 거래" 처리 증명이 본체
- **어필**: 최상 — 면접에서 "PG 장애 나면?"에 구현물로 답하는 유일한 방법

## 2. 가상계좌 + 웹훅 신뢰성 심화 ⭐ 면접 최다 질문 영역

### 왜 만드나
- "웹훅이 두 번 오면? 안 오면? 순서가 뒤집히면?"은 현직자가 가장 파고드는 지점
- 토스페이먼츠 **테스트 키로 입금 시뮬레이션까지 완주 가능** — 실계약 없이 되는 심화 기능

### 실무 구조 (토스페이먼츠 문서 실측)
- 상태 흐름: `READY → WAITING_FOR_DEPOSIT → DONE / CANCELED / EXPIRED`
- 입금기한: `validHours`/`dueDate`, 최대 90일, 기본 7일
- 입금 전 취소는 **전액취소만** 가능. 입금 후 환불은 환불계좌(`refundReceiveAccount`)로 ~2영업일
- 웹훅 재전송: 최대 7회, 지수 백오프 1→4→16→64→256→1024→4096분 (약 3일 19시간). 10초 내 200 응답 요구
- 웹훅 검증: secret 대조 방식(HMAC 아님) + IP 화이트리스트 → **폴링 대사 배치(일 1회 전체 대조)** 병행이 표준

### 핵심 함정 2개 (문서를 깊게 읽어야만 아는 것)

**함정 1 — EXPIRED 시 웹훅이 전송되지 않는다 (문서 명시)**
```
만료 처리는 자체 스케줄러 필수:
dueDate 저장 → 주기 스캔 → 조회 API로 대조 → 주문 취소 + 재고 복원 + 쿠폰/포인트 원복

레이스 컨디션: 만료 처리하는 순간 입금 웹훅이 도착할 수 있다
→ 조건부 UPDATE(WHERE status = 'WAITING_FOR_DEPOSIT') / 낙관적 락으로 해결
```

**함정 2 — DONE → WAITING_FOR_DEPOSIT 역전이**
- 일부 은행(신한 등)에서 입금 실패인데 DONE 통보가 먼저 오고, **최대 2분 후 되돌리는 통보**가 온다. "2분 지연통보" 옵션도 존재
- 상태머신에 역전이를 허용 전이로 등록하고, 이미 발송한 "결제 완료" 후속 처리(알림, 포인트 적립)를 보상하는 로직 필요

### 에스크로 (구매안전서비스)
- 전자상거래법상 **현금성 결제(가상계좌·계좌이체)는 구매안전서비스 의무**, 카드는 선택
- 플로우: `useEscrow: true` + escrowProducts → 결제 통보 후 **영업일 3일 내 배송정보 등록 의무** → 배송완료 후 **영업일 3일 미선택 시 자동 구매확정**. 구매확정 전 부분취소 불가
- 실계약·심사가 필요해 테스트 완주는 어려움 → **도메인 모델링 + 자동 구매확정 배치**로 재현

### 구현 체크리스트
- [ ] 가상계좌 발급 + 입금 웹훅(`DEPOSIT_CALLBACK`) 처리 (멱등)
- [ ] 만료 배치 (dueDate 스캔 → 조회 API 대조 → 원복) + 만료-입금 레이스 테스트
- [ ] DONE 역전이 상태머신 + 후속 처리 보상
- [ ] 입금 후 환불 (환불계좌 플로우)
- [ ] 에스크로 도메인 모델 + 자동 구매확정 배치
- [ ] 웹훅 + 일 1회 폴링 전체 대조 배치

### 평가
- **난이도**: 중 (에스크로는 중~상, 실계약 제약으로 모델링 중심)
- **어필**: 최상 — "EXPIRED 웹훅이 없어서 배치로 보완했다", "DONE 역전이를 처리했다"는 문서를 깊게 읽은 사람만 가능한 발언

## 3. 어드민/백오피스 — 운영 감각의 가장 확실한 증거

### 왜 만드나
- **채용공고 실증**: 스마트로 "PG 정산 배치 및 백오피스"(wanted 163633), 섹타나인 "[PG 시스템 Backend] 정산 및 백오피스"(wanted 210814), 나이스페이먼츠 "PG BO서비스 개발" — **백오피스가 직무명으로 실재**하며 PG사에선 정산 배치와 묶여 나옴
- 대부분 포트폴리오가 happy path에서 끝나는데, 어드민은 "결제는 만들고 나서가 진짜"라는 운영 감각의 증거. 실무자가 매일 쓰는 기능이라 공감도 최고

### 필수 기능
1. **결제 조회/검색**: orderId·paymentKey·기간·상태·금액 복합 검색 (QueryDSL + 커서 페이지네이션 — 성능 소재로도 활용)
2. **강제 취소**: PG 취소 API + 내부 원장 보정 + 현금영수증 연쇄 취소를 **하나의 흐름**으로. 사유 필수 입력, 이중 취소 방지(멱등)
3. **수기 대사 처리**: 자동 대사가 확신 못한 불일치 건 목록 → 사람이 매칭 확정/보정하는 화면 (Modern Treasury의 "결정적 매칭 + 예외만 사람" 원칙의 UI 구현)
4. **웹훅 재발송/수동 동기화**: 토스 대시보드에도 "다시 시도" 버튼 존재 — 우리 어드민에도 수신 누락 건 조회 API 강제 동기화 버튼
5. **DLQ 재처리**: 실패 메시지 조회 + 수동/일괄 재처리
6. **CS 지원**: 사용자별 결제 타임라인(상태 전이 이력 그대로 노출), PG 에러코드 표시, 매출전표 URL 재발급

### 고급 (여기서 급이 갈림)
- **Maker-checker (2인 승인)**: 강제취소·수기 보정 같은 위험 행위는 요청자와 승인자 분리 — 금융 백오피스 표준 패턴
- **개인정보 마스킹 + 언마스킹 사유 입력**: 권한별 마스킹, 언마스킹 행위 자체를 감사 로그에 기록 (ISMS 요건)
- RBAC: 우아한형제들은 백오피스 200개 이상을 OSORI 권한 플랫폼으로 관리 — 역할 기반 화면/기능 제어

### 구현 체크리스트
- [ ] 조회/검색 + 결제 타임라인
- [ ] 강제 취소 (원장 보정 + 연쇄 취소 + 멱등)
- [ ] 대사 불일치 처리 화면 (확정/보정/티켓)
- [ ] 웹훅 수동 동기화 + DLQ 재처리
- [ ] maker-checker 승인 플로우
- [ ] 어드민 행위 전체 감사 로그 (who/when/what/why)

### 평가
- **난이도**: 중 — 개별 기능은 CRUD지만 강제취소·수기 대사의 원장 정합성이 본체. maker-checker까지면 중상
- **어필**: 상 — 채용 직무와 직접 매칭

## 4. 복합결제 — 분산 트랜잭션의 "도메인 필연" 사례

### 왜 만드나
- 억지 MSA가 아니라 **도메인이 강제하는 Saga**: 내부 포인트 차감과 외부 카드 승인은 물리적으로 한 트랜잭션이 될 수 없다
- 무신사 공고 우대사항 "쿠폰/포인트 등 benefit 시스템 업무 경험", 배민 "PG연동, 간편결제, **포인트시스템**"과 직결
- 공개 정리 자료가 드문 영역이라 희소성 큼

### 실무 구조

**포인트+카드 결제 순서 (원칙: 롤백이 확실한 것부터 선점)**
```
1. 포인트 차감 (내부 DB — 실패 시 그냥 롤백, 확실함)
2. 카드 승인 (외부 PG)
3. 카드 실패 → 포인트 복원 (보상 트랜잭션)
4. 보상 자체가 실패할 수 있음 → 보상 요청을 이벤트 테이블에 선기록 + 재시도 큐 + DLQ
```

**부분취소 환불 우선순위 = 포인트 먼저 (면접 대화 유발력 최고)**
1. 카드부터 환불하면 **무상 포인트를 현금화하는 어뷰징** 가능 (10,000원 = 포인트 3,000 + 카드 7,000 결제 → 7,000원 부분취소를 카드로 받으면 포인트가 현금이 됨)
2. 포인트 복원은 내부라 **즉시·확실**, 카드 부분취소는 카드사 의존(3~5일, 일부 카드사 부분취소 미지원 — 카페24 문서 확인)
3. 회계상 카드 역분개 최소화
- 수단별 취소 가능 잔액 추적 (누적 부분취소 ≤ 승인금액) + 취소 멱등키

**쿠폰 안분 (정산 금액 배분)**
- 판매자 부담 쿠폰은 정산금에서 차감 (토스쇼핑 공식: "부분취소 시 쿠폰 비용도 비례하여 취소"), 플랫폼 부담 쿠폰은 수수료에서 조정 (오늘의집 정산 가이드)
- 주문 단위 쿠폰의 라인별 안분:
```
라인 배분액 = 쿠폰액 × (라인금액 / 주문합계)
+ 원 단위 절사
+ 끝전(단수차)은 마지막 라인에 몰아주기
+ 불변식: sum(라인 배분액) == 쿠폰액   ← 속성 기반 테스트(property-based test)로 검증
```
- 배민 정산팀 공식 요구: "결제수단/보조수단/상품정책에 따라 **1원 오차 없이**"

**참고 설계 — 우아한형제들 포인트 시스템 전환기**
- UPDATE가 아닌 **Insert 기반 원장** 모델, FIFO 차감 (유효기간 임박 포인트부터)
- 포인트 "사용"만 중복 방지를 위해 동기 처리, 나머지는 비동기
- SQS 순서 역전(주문 직후 취소) 시 재발송 5회 → DLQ 수동 검토

### 구현 체크리스트
- [ ] 포인트 원장 (Insert 기반, FIFO 차감, 유효기간)
- [ ] 복합결제 Saga (포인트 선점 → 카드 승인 → 실패 시 보상)
- [ ] 실패 매트릭스 전수 테스트 (승인 성공 후 확정 실패, 보상 실패 재시도 등)
- [ ] 부분취소 환불 우선순위 엔진 + 수단별 취소 가능 잔액
- [ ] 쿠폰 안분 순수 함수 + 속성 기반 테스트 ("어떤 조합에도 합계 불변")

### 평가
- **난이도**: 중~상 — 해피패스는 중, 실패 매트릭스가 본체. 쿠폰 안분은 순수 함수라 테스트하기 좋음
- **어필**: 상 — "왜 포인트부터 환불하나요?"에 어뷰징 방지 논리로 답하면 도메인 감각 강하게 어필

## 5. 빌링키 정기결제(구독) — 연동이 아니라 "실패 설계"가 본체

### 왜 만드나
- **토스페이먼츠는 스케줄링을 제공하지 않는다** — 결제 주기 배치, 실패 처리, 상태머신 전부 가맹점 몫. 공식 예제(node-cron)조차 실패 처리는 `.catch()` 로깅 수준 → 여기가 차별화 지점
- "결제 실패하면 바로 해지하나요?"는 구독 도메인 면접 단골

### 실무 구조

**빌링키 발급 (토스페이먼츠 공식)**
- 빌링키 = 카드번호·유효기간·CVC를 암호화한 토큰. 최초 1회 인증 후 매 주기 무인증 결제. 자동결제는 별도 리스크 심사·계약 필요(테스트 키는 무관)
- 발급 2방식: ① **결제창(authKey) 방식** — SDK `requestBillingAuth()` → `authKey` 수신 → 발급 API 호출. **카드정보가 가맹점 서버를 안 거침** (권장) ② API 방식 — 카드번호 직접 수집(보안 요건 상승)
- `customerKey`: **UUID 같은 무작위 값 필수** (이메일·자동증가 ID 금지). 빌링키+customerKey **이중 키 구조**라 빌링키 단독 유출로는 결제 불가
- 승인: `POST /v1/billing/{billingKey}`, 삭제: `DELETE /v1/billing/{billingKey}`

**Dunning — 실패 회수 전략 (Stripe/Recurly 벤치마크)**

| 실패 유형 | 예시 | 대응 |
|---|---|---|
| **Hard decline** | 도난/분실/무효 카드 | **재시도 금지** — 반복 시도하면 카드사에서 가맹점 평판 점수 하락. 즉시 고객에게 카드 변경 요청 |
| **Soft decline** | 잔액부족, 한도초과, 일시 오류 | 재시도 유효. Stripe 기본 "2주 내 8회", 회수 대부분은 실패 후 2~7일 내. 잔액부족이 회수율 최고 |

- Stripe Smart Retries: ML이 국가/카드별 성공 시간대 학습 (예: 자정 직후 한도 리셋으로 00:01 성공률 상승)
- 국내 카드사별 최적 재시도 시점 공개자료 없음 → **"실패 코드 기반 분기 + 시간대 가설(자정 직후, 급여일 25일)" 설계 + 가설을 문서화**하는 것이 현실적이고 오히려 어필
- 에러 유형별 재시도 간격 차등 (Recurly: Gateway Error 2일, Issuer Unavailable 3일)
- 최종 실패 시 구독 상태 3택: canceled / unpaid / past_due. 실측 회수율 25~35%

**구독 상태머신 (Google Play 표준 레퍼런스)**
```
ACTIVE → IN_GRACE_PERIOD (7일, 접근 유지·갱신일 불변)
       → ON_HOLD (53일, 접근 차단·복구 시 갱신일 이동)
       → EXPIRED
CANCELED: 기간말 해지가 기본 (expiryTime까지 접근 유지)
```

**Proration (플랜 변경 일할계산 — Stripe 방식)**
- 변경 시점 기준 (구 플랜 미사용분 크레딧) + (신 플랜 잔여기간 청구), **초 단위 계산**
- 업그레이드 = 즉시 차액 청구 / 다운그레이드 = 크레딧 적립 (현금 환불 없음)
- Preview API로 미리보기 → 같은 proration_date로 실행 (미리보기와 실제 금액 불일치 방지)
- 월말 엣지: 31일 갱신이 2월을 만나면 28일로 고정

### 구현 체크리스트
- [ ] 빌링키 발급 (결제창 방식) + customerKey UUID 정책
- [ ] 결제 주기 배치 (멱등 — 같은 주기 중복 청구 방지)
- [ ] 실패 코드 분류 테이블 + dunning 스케줄러 (hard/soft 분기)
- [ ] 구독 상태머신 (유예기간 → 정지 → 만료) + 상태별 접근 제어
- [ ] proration 계산기 (순수 함수 + 엣지 케이스 테스트: 월말, 같은 날 왕복 변경)
- [ ] 미리보기 API

### 평가
- **난이도**: 빌링키 연동 하~중 / dunning+상태머신+proration까지 중~상 (금액 계산 엣지케이스 다수)
- **어필**: 단순 "빌링키→cron→승인"은 튜토리얼 수준(중). **dunning 분기 + 유예기간 상태머신 + proration이 결합돼야 상**

## 6. 선불충전 월렛(페이머니) — 규제를 코드로 반영

### 왜 만드나
- 카카오페이머니·토스머니·네이버페이머니가 전부 이 구조 — 페이사 지원 시 직결
- 잔액 정합성/이중지불/락 전략은 결제 면접 단골
- **개정 전금법을 도메인 규칙으로 코드에 반영하면 "규제를 이해하는 개발자"로 차별화**

### 실무 구조

**법적 프레임 (전자금융거래법)**
- 선불전자지급수단 **발행한도: 무기명식 50만원, 기명식 200만원** (금융위 유권해석 — 카카오페이머니 200만원 한도의 근거)
- **2024.9.15 개정 전금법**: 등록 면제기준 축소, **선불충전금 전액 별도관리 의무**(은행 예치/신탁/보증보험), 안전자산 운용 한정, BNPL 법제화(월 30만원 한도)
- 포트폴리오 반영: 충전 시 한도 검증 규칙, "별도관리 계정" 을 원장의 독립 계정으로 모델링

**원장 설계 (03 문서의 복식부기 그대로)**
- 계정/거래/분개 3테이블, 모든 이동 = 차변+대변 쌍, append-only
- **잔액 컬럼 UPDATE식 지갑은 어필 제로 (오히려 감점)** — 잔액은 엔트리 합산 파생값 + 스냅샷 캐시

**동시성 (Modern Treasury 공식 패턴)**
- `lock_version` 낙관적 락 + **잔액 조건부 커밋** (`balance >= 0`)으로 마이너스 잔액 차단
- **핫 어카운트 패턴**: 사용자 계정은 동기 락(충돌 드묾), 고처리량 정산/수수료 계정은 **비동기 배치 집계** (모든 거래가 만지는 계정에 동기 락을 걸면 그 계정이 전체 병목)
- 토스뱅크 사례: Redis 글로벌 락 + DB row lock 병행, 비핵심 작업 Kafka 분리

### 구현 체크리스트
- [ ] 충전/차감/환불 원장 (복식부기, append-only)
- [ ] 전금법 한도 검증 (기명 200만원) — 규칙의 법적 근거를 ADR에 인용
- [ ] 별도관리 계정 모델링 (충전금 ↔ 회사 자금 분리)
- [ ] 낙관적 락 + 조건부 커밋 + 핫 어카운트 비동기 집계
- [ ] **정합성 증명 테스트**: 동시 차감 100건 → 이중차감·마이너스 잔액 0건 입증

### 평가
- **난이도**: 중 (2~4주). 코드량이 아니라 **정합성 증명**이 핵심
- **어필**: 상 — 규제 반영 + 동시성 증명 조합

## 7. 필드 암호화 + 감사 로그 — 법령 수치를 근거로

### 왜 만드나
- 카카오페이 우대사항 "Secure Coding 생활화·보안 취약점 사전 파악"
- "법 고시에서 카드번호·계좌번호가 암호화 의무 대상", "전금법 5년 보존" 같은 **규제 수치를 근거로 설계했다고 쓰면 금융권 서류에서 눈에 띔**

### 실무 구조

**필드 암호화**
- 「개인정보의 안전성 확보조치 기준」상 암호화 의무 대상: 주민번호·여권번호·운전면허번호·외국인등록번호·**신용카드번호·계좌번호**·바이오정보. 비밀번호는 일방향 해시
- 알고리즘: AES-256-**GCM** (CBC 아님 — 인증 태그로 변조 감지)
- **Envelope encryption (키 관리의 정석)**:
```
데이터 → DEK(Data Encryption Key)로 암호화
DEK → KMS의 마스터키(KEK)로 암호화해 데이터와 함께 저장
키 로테이션: KMS 자동 로테이션 시 이전 키 버전 보존 (재암호화 불필요)
```
- Spring 구현: JPA `AttributeConverter` + `@Convert`. **블로그 예제 대부분이 키 하드코딩(안티패턴)** → KMS/Vault 로딩 + 키 버저닝이 차별화. LocalStack KMS로 무료 재현 가능
- 암호화 컬럼은 검색 불가 → **blind index(HMAC 해시 컬럼)** 병행 (같은 평문 = 같은 해시로 equality 검색만 허용)

**감사 로그 (Audit Trail)**
- 법적 근거: 전자금융거래법 제22조 — **거래기록 5년 보존**. 전자금융감독규정 — 원장 직접 접근 기록 5년, 시스템 가동기록 1년 이상
- 구현 3패턴:
  1. **Hibernate Envers** (`@Audited` → _AUD/REVINFO 테이블 자동) — 저비용
  2. **도메인 이벤트 기반 append-only** — 결제 상태 전이 이력(Phase 1)과 궁합 최고
  3. DB 트리거/CDC — 앱 우회 DML까지 포착
- 강화: INSERT-only + **DB 계정 권한에서 UPDATE/DELETE 회수**, 고급은 해시 체인(위변조 감지)

**PCI-DSS (코드가 아닌 문서화 영역)**
- 12개 요구사항 (네트워크 보안, 저장 데이터 보호, 전송 암호화, need-to-know 접근, MFA, 로깅·감사추적, 정기 테스트 등)
- **국내 가맹점이 직접 심사 안 받는 이유**: PG 결제창/빌링키를 쓰면 카드데이터가 가맹점 서버를 안 지나감 → PCI 의무는 PG/VAN 부담 (구조적으로 SAQ A — 13항목 수준, 직접 다루면 SAQ D 300+ 항목)
- **README에 "우리 시스템의 PCI scope 분석" 1페이지**로 녹이면 강함: "우리는 카드번호를 저장하지 않도록 결제창 방식을 선택했고, 따라서 scope가 이렇게 축소된다"

### 구현 체크리스트
- [ ] AttributeConverter 기반 AES-256-GCM 필드 암호화 (계좌번호 등)
- [ ] LocalStack KMS + envelope encryption + 키 버저닝
- [ ] blind index 검색
- [ ] 감사 로그 (Envers 또는 이벤트 기반) + UPDATE/DELETE 권한 회수
- [ ] README: PCI scope 분석 1페이지 + 법령 근거 ADR

### 평가
- **난이도**: 암호화 중 (Converter 반나절, envelope+로테이션+blind index로 중 상위), 감사 로그 하~중, PCI 문서화 하
- **어필**: 상 — 금융권 지원 시 급상승

## 8. 운영 고도화 — "결제 로직 무중단 교체" 스토리

### 왜 만드나
- 리플레이 검증은 주니어 포트폴리오에서 거의 안 나옴 (SLASH 24 세션 주제일 정도로 실무 고급 기법)
- 단, **전환할 대상(v1 레거시)이 프로젝트에 실재해야** 작위적이지 않음 — Phase 1에서 만든 결제 로직을 Phase 5 개선 버전으로 교체하는 시나리오가 자연스러움

### 실무 구조 — 4요소를 하나의 스토리로

**① 병렬 비교 검증 (토스뱅크 코어뱅킹 전환기 방식)**
- 신구 로직 **병렬 호출 + 결과 comparator + 불일치 메트릭/알림**
- Staging 배치 대량 검증 + 엣지 케이스 E2E + 트래픽 비율 점진 전환
- 축소 재현: 결제 금액 계산/정산 로직 v1·v2 병렬 실행, 응답은 v1으로 하되 diff를 메트릭으로 적재

**② Feature Flag (카카오페이 자체 개발기 참고)**
- 카카오페이 구조: 어드민/서빙 분리, Redis Pub/Sub + 로컬 캐시 + 폴링 백업
- 결제 특화 논점: **사용자 해시 버킷팅으로 1% → 100% 점진 이동 + 진행 중 거래의 경로 고정** (한 거래가 v1으로 시작했으면 끝까지 v1) + 멱등성 유지
- 가성비 최고 (구현 하~중인데 스토리 가치 큼). 오픈소스는 OpenFeature(11번가 사례)

**③ 섀도 트래픽/카나리**
- 토스페이먼츠 SDK 카나리: Cohort 헤더 + 쿠키 고정(동일 사용자 동일 버전), weight 변경 즉시 반영 = 실시간 롤백
- **결제 도메인 치명 주의점: 섀도 경로의 PG 호출은 스텁/기록 모드로 — 안 그러면 이중 승인 사고**. 이 설계 포인트를 문서화하는 것 자체가 어필

**④ 카오스 엔지니어링**
- 올리브영 사례: 인프라가 아닌 **애플리케이션 레벨** — mitmproxy로 API 응답 필드에 null 주입
- 결제 적용: **Toxiproxy를 mock PG 앞에 두고 지연/단절 주입 → 서킷브레이커·미확정 거래 대사가 견디는지 가설-실험-결과 리포트**
- **문서화 없으면 도구 튜토리얼로 평가절하** — 반드시 "가설 → 실험 → 관측 → 결론" 형식

**⑤ (선택) 무중단 스키마 변경**
- expand-contract 패턴: 새 컬럼 추가 → 이중 쓰기 + 백필 → 구 컬럼 제거
- 도구: pt-osc(트리거 기반, 데드락 리스크) vs **gh-ost**(binlog 기반, FK 미지원)
- 어필 조건: **수백만 row + 쓰기 트래픽이 흐르는 상태에서 실측** (소요 시간, 복제 지연 기록)

### 구현 체크리스트
- [ ] feature flag 모듈 (해시 버킷팅 + 거래 경로 고정)
- [ ] v1/v2 병렬 비교 실행기 + 불일치 메트릭 대시보드
- [ ] 점진 전환 시나리오 실행 (1→10→50→100%) + 즉시 롤백 데모
- [ ] Toxiproxy 카오스 실험 리포트 2~3건
- [ ] (선택) gh-ost로 결제 테이블 스키마 변경 실측

### 평가
- **난이도**: flag 하~중(가성비 최고), 병렬 검증 중, 카오스 중
- **어필**: **"flag + 병렬 비교 + 점진 전환 + 즉시 롤백"을 하나의 '무중단 교체' 스토리로 묶으면 최강**. 낱개로는 조건부

## 9. 이상거래탐지(FDS) — 룰 기반으로 충분, 판단이 어필

### 왜 만드나
- 결제사 채용공고에 FDS가 실제 명시됨 (섹타나인 "Risk Management 시스템 설계")
- 어필 핵심은 탐지 정확도가 아니라 **아키텍처 판단** — 동기 인라인 판정 vs 비동기 사후 탐지, 무배포 룰 변경

### 실무 구조
- 3요소: 정보수집 → 분석 → 대응
- **대표 룰**:
  - **Velocity check**: 계정/카드/기기/IP 키별 단위시간 시도 횟수 (Redis 카운터 or Kafka Streams 윈도우)
  - 금액 이상치: 30일 평균 대비 배수
  - 동일 카드 다계정, 신규 기기/해외 IP, 블랙리스트
- **스코어링 → 4단계 대응**: 룰별 가중치 합산 → risk score 구간별 `ALLOW / CHALLENGE(추가인증) / BLOCK / REVIEW(사후 심사)`
- **룰은 코드가 아닌 데이터(DB/yaml)로 관리 → 무배포 변경** — 이 설계가 핵심 어필
- 스트림 처리: Kafka account_id 파티셔닝 → **슬라이딩 윈도우 5~10분** 집계. **포트폴리오 규모에선 Flink 과잉, Kafka Streams로 충분** — 이 선택 근거를 ADR로 남기는 것 자체가 포인트
- **동기 vs 비동기 판단**: 결제 경로 인라인 판정은 지연 예산(예: 50ms) 내 룰만, 무거운 분석은 비동기 사후 탐지 + 사후 BLOCK
- ML은 넣지 말고 **스코어러 인터페이스만 설계** (확장점 표시)

### 실무 레퍼런스 규모 (아는 척 포인트)
- 카카오페이: Hadoop → Airflow 피처 생성 → 피처 스토어(HBase 배치 + Redis 준실시간) → MLflow → Triton 서빙. 동적 피처로 신규 사고 유형 탐지율 30% 개선, 월 71만건 감지
- 카카오뱅크: 기존 룰 엔진에 Kafka → Flink → Redis 피처스토어 → ML을 **병렬 추가** (룰 대체가 아님). 룰 한계: 신종 사기, 사기꾼의 룰 학습 회피
- 토스: 전 송금 실시간 ML 추론, 2022년 31만건 차단

### 구현 체크리스트
- [ ] 룰 정의 테이블 (조건/가중치/활성화 — 무배포 변경)
- [ ] velocity 카운터 (Redis) + Kafka Streams 슬라이딩 윈도우 집계
- [ ] 스코어링 엔진 + 4단계 대응 (CHALLENGE는 mock 추가인증)
- [ ] 동기 인라인(경량 룰) / 비동기 사후(중량 룰) 분리 + 지연 예산 측정
- [ ] REVIEW 큐 → 어드민(3번) 연동

### 평가
- **난이도**: 룰엔진 + Redis velocity면 중 (2~3주), Kafka Streams 윈도우 + 동적 룰까지면 상 경계
- **어필**: 상 — **단순 if문 몇 개는 역효과** (도메인을 얕게 봤다는 신호)

## 10. 현금영수증 + 증빙 3종 — 순수 국내 도메인 완결성

### 왜 만드나
- 해외 자료로 못 배우는 **순수 국내 도메인** → "국내 실무를 안다"는 신호
- 기술이 아닌 **법적 의무**라 커머스 결제 오픈 체크리스트에 항상 포함
- 가성비 좋은 서브 기능 (난이도 하~중)

### 실무 구조

**증빙 3종 자동 결정 로직 (커머스 결제 필수 요건)**
| 결제수단 | 법정 증빙 | 비고 |
|---|---|---|
| 카드 | **매출전표** | 토스 Payment 객체 `receipt.url` 자동 제공 — 저장·노출만 하면 CS "영수증 주세요"의 90% 해결 |
| 가상계좌·계좌이체 (현금성) | **현금영수증** | "가상계좌 = 현금거래 = 현금영수증 대상"이 핵심 도메인 지식 |
| B2B | **세금계산서** | 카드 결제 건은 매출전표가 증빙 — **세금계산서 중복 발행 금지** |

**현금영수증 (토스페이먼츠 API 실측)**
- 의무발행업종은 10만원 이상 현금거래 시 요구 없어도 발급 의무, 위반 시 **미발급액 20% 가산세**
- `POST /v1/cash-receipts` (type: 소득공제|지출증빙, customerIdentityNumber, taxFreeAmount)
- **발급은 비동기**: `IN_PROGRESS → COMPLETED / FAILED` — 상태 추적 필요 (여기도 상태머신)
- 부분취소 가능: `POST /v1/cash-receipts/{receiptKey}/cancel`
- **운영 함정 = 최대 어필 포인트**: 수동 발급 API 건은 **결제 취소 시 가맹점이 직접 현금영수증 취소 API를 호출**해야 함 (실무 사고 다발 지점) → **"결제 취소 → 현금영수증 자동 연쇄 취소"** 구현이 면접 대화 유발력 최고
- 가상계좌는 발급 요청에 cashReceipt 객체를 넣으면 입금 시 자동 발급

**세금계산서**
- 법인 전체 + 개인 8천만원 이상 전자발행 의무
- 자체 시스템은 홈택스 직결이 아니라 **팝빌/바로빌 ASP Open API 연동이 표준** (샌드박스 제공)
- 결제 시스템에서의 위치: B2B, 특히 "플랫폼 → 셀러 수수료 세금계산서"로 **정산의 마지막 단계** — 정산 시스템(Phase 4)이 있을 때만 2순위로

### 구현 체크리스트
- [ ] 증빙 자동 결정 라우터 (결제수단 → 증빙 종류)
- [ ] 현금영수증 발급 (비동기 상태 추적) + 실패 재시도
- [ ] **결제 취소 → 현금영수증 연쇄 취소** (부분취소 포함)
- [ ] 매출전표 URL 저장·노출 + 어드민 재발급
- [ ] (선택) 팝빌 샌드박스 세금계산서 — 정산 수수료 명세 연동

### 평가
- **난이도**: 현금영수증 하~중 (연쇄 취소 + 비동기 추적까지면 중), 매출전표 하, 세금계산서 중
- **어필**: 중~강 (국내 한정 매우 강함). 단독으론 약하고 **"증빙 서브도메인"으로 묶으면** 강함

## 로드맵 연결 — Phase 7~8 (선택 확장)

기본 로드맵(06 문서) 완료 후, **전부 하지 말고 지원 타깃에 맞춰 2~3개 선택**:

| **핀테크/페이** (카카오페이, 토스, 네이버페이) | ⑥ 월렛 + ⑦ 암호화·감사 로그 + ⑨ FDS | 선불업 라이선스 도메인 + 금융 규제 + 리스크 |
| **구독 서비스** (콘텐츠/SaaS) | ⑤ 빌링키+dunning + ⑧ 운영 고도화 | 반복 결제 신뢰성이 곧 매출 |

### 기존 기반과의 결합 지점

| 확장 기능 | 얹히는 기존 기반 (Phase) |
|---|---|
| 멀티 PG 라우팅 | 서킷브레이커 + 3-상태 모델 (Phase 2) |
| 가상계좌 대사 | 대사 엔진 (Phase 4), 웹훅 파이프라인 (Phase 3) |
| 어드민 | 대사 예외 큐 (Phase 4), DLQ (Phase 3), 감사 이력 (Phase 1) |
| 복합결제 | 보상 트랜잭션 (Phase 2), 원장 (Phase 4) |
| 구독 dunning | 재시도 전략 + 복구 배치 (Phase 2) |
| 월렛 | 복식부기 원장 (Phase 4), 동시성 제어 (Phase 5) |
| FDS | Kafka 파이프라인 (Phase 3), 관측성 (Phase 6) |
| 무중단 교체 | 부하테스트 (Phase 5), 메트릭 (Phase 6) |

> **새 기능이 아니라 기존 기반의 확장으로 설계하면 시스템의 일관성 자체가 어필이 된다.** 면접에서 "멀티 PG를 어떻게 붙였나요?" → "Phase 2의 서킷브레이커 위에 어댑터만 추가했습니다"라고 답할 수 있는 구조가 목표.
