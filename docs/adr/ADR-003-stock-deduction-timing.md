# ADR-003. 재고는 결제 승인 성공 시점에 차감한다

- 상태: 채택 (Accepted)
- 날짜: 2026-07-05
- 관련: Phase 1(주문/결제 코어), Phase 2(보상 트랜잭션)

## 맥락

주문 → 결제 흐름에서 재고를 언제 차감할지 정해야 한다. 두 시점이 있다.

- **주문 생성 시 선점**: 주문을 만들 때 재고를 미리 잠근다.
- **승인 성공 시 차감**: 결제가 실제로 승인된 뒤에만 재고를 뺀다.

## 결정

**결제 승인이 성공한 시점에 차감**한다.

```
주문 생성 → (재고 여유 확인만) → 결제 승인 → 재고 차감 → 완료
```

## 근거

1. **재고 잠김(dangling reservation)이 없다**: 선점 방식은 결제까지 안 가고 이탈한 주문이 재고를 물고 있어, 만료·해제 배치를 또 만들어야 한다. 승인 시 차감은 "돈이 실제로 움직인 것"에만 재고를 소비한다.
2. **구현이 단순**: 상태가 하나 줄고(선점 상태 불필요), 예약-해제 로직이 사라진다.
3. **실패 설계와 자연스럽게 연결**: 승인은 됐는데 재고 차감이 실패하는 경우(품절 경합) → **망취소/보상 트랜잭션**([Phase 2])으로 결제를 되돌린다. 이건 어차피 만들어야 하는 핵심 기능이라, 재고 차감을 그 위에 얹으면 별도 메커니즘이 필요 없다.

## 트레이드오프 / 포기한 것

- **승인 후 품절 가능성**: 승인이 성공했는데 그 찰나에 재고가 0이 될 수 있다. 이때는 승인을 보상(망취소)하고 사용자에게 품절 안내. 선점 방식이었다면 이 창을 없앨 수 있지만, 대신 재고 잠김이라는 상시 비용을 진다.
- **선착순/한정판**처럼 "승인 전에 반드시 재고를 확보"해야 하는 시나리오에는 부적합. 그 경우는 Phase 5에서 **Redis 대기열 + 선점**을 별도 전략으로 다룬다(같은 코드에 두 전략을 공존시켜 부하테스트로 비교).

## 후속

- Phase 1: 승인 성공 시 재고 차감. 차감 실패 시 일단 예외 → Phase 2에서 보상으로 승격.
- Phase 5: 재고 차감 동시성 실험(비관적 락 vs 낙관적 락 vs Redisson)의 대상 테이블로 재사용.
