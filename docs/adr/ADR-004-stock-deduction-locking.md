# ADR-004. 재고 차감은 조건부 UPDATE 전략을 쓴다

- 상태: 채택 (Accepted)
- 날짜: 2026-07-06
- 관련: [ADR-003](ADR-003-stock-deduction-timing.md), Phase 5(성능)

## 맥락

재고 차감은 동시 요청이 몰리는 지점이다(선착순·인기 상품). 음수 재고·초과 판매(oversell)를 막으면서 성능도 지켜야 한다. 후보 3종:

- **비관적 락**: `SELECT ... FOR UPDATE`로 행을 잠그고 차감
- **낙관적 락**: `@Version` + 충돌 시 재시도
- **조건부 UPDATE**: `UPDATE ... SET quantity = quantity - :n WHERE quantity >= :n` (락 없이 원자적)

## 실험

`StockLockComparisonTest`에서 실제 스레드로 동시 차감을 측정했다(H2, 30스레드가 재고 20에 동시 차감).

| 전략 | 초과판매 | 완판 | 소요 |
|---|---|---|---|
| **조건부 UPDATE** | 0 | 20/20 | **8ms** |
| 낙관적 락 | 0 | 20/20 | 17ms |
| 비관적 락 | 0 | 20/20 | 32ms |

세 전략 모두 **안전**(초과판매 0, 최종재고 = 재고 − 성공)했다. 차이는 성능이다.

- **조건부 UPDATE가 최속**(8ms). 락도 재시도도 없이 DB 한 번의 원자적 UPDATE로 끝난다.
- 비관적 락은 행 잠금·트랜잭션 왕복으로 가장 느리다(32ms). 커넥션도 오래 점유한다.
- 낙관적 락은 중간(17ms). 단, **고경합에서는 재시도 소진으로 정상 차감도 실패(미달판매)**할 수 있다(스레드 수를 150으로 올리면 재현됨). 재시도 폭증 자체가 부하다.

## 결정

기본 전략으로 **조건부 UPDATE**를 채택한다. `CheckoutService`가 승인 성공 시
`StockDeductionService.deductConditional`을 호출한다.

## 근거 / 트레이드오프

- 단일 자원(재고 행) 차감에는 조건부 UPDATE가 가장 저비용·안전하다.
- 비관적 락은 "차감 전후에 여러 읽기/계산이 필요"한 복잡한 시나리오에서만 가치가 있다. 단순 차감엔 과하다.
- 낙관적 락은 충돌이 드문 자원(사용자별 포인트 등)에 적합하다. 인기 상품 재고처럼 한 행에 몰리는 hot row에는 부적합.
- **멀티 인스턴스에서 DB 부하를 분산**해야 하면 Redis 분산락(Redisson)을 추가로 고려한다(확장 08 문서). 이 프로젝트는 단일 DB 기준으로 조건부 UPDATE가 정답.

## 후속

- 세 전략 구현(`StockDeductionService`)은 비교 근거로 코드에 남겨 둔다.
- 실제 애플리케이션 레벨 부하테스트(k6)는 `docs/performance/`에 방법론과 스크립트로 정리한다.
