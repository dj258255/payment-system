# ADR-007. 체크아웃은 단일 트랜잭션을 유지한다(외부 PG 호출을 포함해서)

- 상태: 채택 (Accepted)
- 날짜: 2026-07-07
- 관련: [ADR-003](ADR-003-stock-deduction-timing.md), [ADR-001](ADR-001-architecture-spring-modulith.md), Phase 2(실패 처리)

## 맥락

`CheckoutService`는 클래스 레벨 `@Transactional`이고, `confirm()` 한 트랜잭션 안에서 **포인트 선점 → PG 승인(외부 HTTP) → 재고 차감 → 주문 상태 전이**를 처리한다. 즉 **외부 PG 호출이 DB 트랜잭션 안**에 있다.

이건 널리 알려진 안티패턴이다 — "**외부 API 콜 중 DB 커넥션을 붙잡지 마라**". 커넥션을 외부 응답까지 점유하면, PG가 느려질 때 커넥션 풀이 고갈되어 전면 마비로 번진다. 실제 장애 사례가 많다(다운스트림 지연으로 인한 연쇄 결제 실패, 느린 저장프로시저가 커넥션을 소진시킨 수 시간 outage, 트래픽 2배에 ORM 최대 커넥션 고갈 등). **서킷브레이커도 "기다리는 동안 풀이 찬다"** — 완전히 막지 못한다.

고규모/마이크로서비스의 정석은 이 흐름을 **사가(saga)로 쪼개는** 것이다: 짧은 로컬 트랜잭션(예약) → 외부 콜(트랜잭션 밖) → 확정/보상(짧은 트랜잭션). 각 단계는 커넥션을 외부 콜 동안 붙잡지 않는다.

## 결정

**이 시스템(모듈러 모놀리스, 단일 DB, 바운디드 스케일)에서는 체크아웃을 단일 트랜잭션으로 유지한다.** 안티패턴을 인지하되, 아래 트레이드오프 분석에 따라 의도적으로 이 설계를 택한다.

## 근거 / 트레이드오프

**단일 트랜잭션이 주는 것 — 진짜 ACID 원자성.** 포인트 차감·결제 레코드·재고·주문 상태가 **함께 커밋되거나 함께 롤백**된다. 외부 부수효과(PG 승인)만 롤백 불가라 보상이 필요하고, 그것도 이미 처리한다: 승인 후 재고 부족 → 망취소 보상 태스크(durable), PG 타임아웃 → `UNKNOWN` 보존 후 복구 배치가 조회로 확정, 재시도 → `paymentKey`가 PG측 멱등키.

**사가로 쪼개면 잃는 것 — 그 원자성.** 사가는 격리(isolation)를 포기한다. Phase 1에서 포인트 예약이 커밋된 뒤 크래시로 Phase 3(확정/보상)를 못 하면, **포인트 예약·결제·주문이 어중간하게 멈춘 사가**로 남는다. 이를 복구하려면 "멈춘 사가를 완결 또는 롤백"하는 별도 회복 경로가 필요하다(포인트만 복원이 아니라 상태 기계 전체). 즉 사가는 **원자성을 커넥션 점유와 맞바꾸고, 그 대가로 새 크래시 엣지케이스와 회복 복잡도**를 떠안는다.

**이 맥락에서의 판단.** "외부 콜을 트랜잭션 밖으로"는 **고규모·다중 인스턴스·느린/원격 PG** 환경에서 가장 강한 규칙이다. 이 시스템은:

- **커넥션 점유 위험이 이미 상한이 걸려 있다** — Hikari `connection-timeout: 3s` + PG 호출 자체의 타임아웃/서킷브레이커로, 느린 PG여도 커넥션을 오래 붙잡지 않고 빠르게 UNKNOWN으로 떨군다. 바운디드 스케일 모놀리스에선 풀 고갈 임계에 잘 도달하지 않는다.
- **단일 DB 모놀리스**라 진짜 ACID 원자성이 저비용으로 손에 들어온다. 이걸 포기하고 사가 복잡도(+ 멈춘-사가 회복)를 떠안는 건 이 규모에선 **과적합**일 수 있다.

따라서 트레이드오프의 무게추는 이 맥락에선 **원자성 유지** 쪽이다. "규칙을 모르고 안 한 것"이 아니라 "규칙과 그 예외를 알고 이 규모에선 원자성을 택한 것"이다.

## 마이그레이션 경로 (스케일이 요구하면)

트래픽이 커지거나 PG 지연이 상시화되어 커넥션 점유가 실측 병목이 되면, 3단계 사가로 이행한다:

1. **Phase 1 (짧은 tx)**: 검증 · `order.startPayment()` · 포인트 예약 · 결제 `IN_PROGRESS` 적재 → 커밋.
2. **Phase 2 (트랜잭션 밖)**: PG 승인. 커넥션을 붙잡지 않는다. (부수 효과: 결제 레코드가 PG 콜 **전에** 커밋돼 있어, 크래시-후-복구의 charge 안전성은 오히려 좋아진다.)
3. **Phase 3 (짧은 tx)**: 결과 반영 — 성공이면 재고 차감·`markPaid`·이벤트 발행, 실패/미확정이면 포인트 복원·주문 복귀.
4. **추가 필수**: 멈춘 사가 회복 — Phase 1과 3 사이에서 죽은 주문(포인트 예약됨 + 결제 IN_PROGRESS)을 스캔해 PG 조회로 완결하거나 예약 자원을 복원하는 배치. 이게 사가 이행의 진짜 비용이다.

이 경계 재설계는 크라운주얼(체크아웃)을 건드리고 ACID 원자성을 사가 복잡도와 맞바꾸는 **최대·최고위험 변경**이므로, 실측 병목이 확인되기 전에는 착수하지 않는다(YAGNI).
