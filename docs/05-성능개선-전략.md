# 05. 성능 개선 전략 — 수치로 증명하는 포트폴리오

> 카카오페이 "온라인 결제 서비스 2.5배 성능 개선기"가 모범 답안이다.
> 핵심은 기술 나열이 아니라 **"목표 수치 → 한계 측정 → 병목 계측 → 개선 → 전후 비교"** 스토리.

## 1. 개선 스토리 프레임 (이 순서 그대로 리포트를 쓴다)

카카오페이 실사례로 보는 프레임:

| 단계 | 카카오페이 실제 사례 |
|---|---|
| ① 목표 수치 산정 근거 | 평시 40~50 TPS → 11월 11일 프로모션 대비 **목표 300 TPS** |
| ② 부하테스트로 현재 한계 측정 | 초기 테스트 결과 **157 TPS 한계** |
| ③ 병목 계측 | APM 계측 → "조회 쿼리가 전체의 98%", 결제 1건에 API 16~20회 호출, OSIV로 긴 트랜잭션 |
| ④ 개선 (한 번에 하나씩, 재측정) | Redis 캐싱(putIfAbsent로 동시성 정합성) → OSIV 제거 → Entity 대신 DTO 프로젝션 → Redis 병목 발견 후 로컬 캐시(2-tier) |
| ⑤ 전후 수치 대비 | **TPS 170 → 400 (2.35배), DB CPU 74% → 50%** + 남은 한계와 다음 계획 |

출처: https://tech.kakaopay.com/post/improve-service-performance/

> **"측정 없이는 개선 없다"** — 개선 항목마다 반드시 전후 수치를 남긴다. 토스페이먼츠 이력서 가이드의 "구체적으로 쓰라"가 정확히 이것.

## 2. 동시성 제어 — 시나리오별 락 선택 (면접 최다 빈출)

### 기법 비교

| 기법 | 원리 | 적합 상황 | 함정 |
|---|---|---|---|
| **낙관적 락** | `@Version` 컬럼, 충돌 시 예외+재시도 | 충돌 드묾 (읽기 많고 쓰기 적음) | 충돌 잦으면 재시도 폭증으로 역효과 |
| **비관적 락** | `SELECT ... FOR UPDATE` | 충돌 빈번 (재고 차감) | 커넥션 오래 점유 → 풀 고갈 위험, 데드락 |
| **Redis 분산락 (Redisson)** | pub/sub 기반 락 (Lettuce 스핀락과 달리 Redis 부하 적음) | 멀티 인스턴스, DB 부하 분산 필요 | **락 해제가 트랜잭션 커밋보다 먼저면 정합성 깨짐** (락 안에서 커밋까지 끝내야) |
| **유니크 제약** | DB가 중복 INSERT 차단 | 중복 결제 방지의 **최후 방어선** | 사후 처리(예외 핸들링) 필요 |

### 시나리오별 선택 (포트폴리오에서 "비교 실험"으로 만들 것)

- **중복 결제(따닥)**: 멱등키 + 유니크 제약이 정석 (락 불필요)
- **재고 차감**: 충돌 빈번 → 비관적 락 vs Redisson 분산락을 **부하테스트로 비교 선택** ← 이 비교 실험 자체가 차별화 콘텐츠
- **포인트 차감**: 한 사용자의 동시 요청은 드묾 → 낙관적 락 + 재시도, 혹은 사용자 단위 분산락
- **참고**: 토스뱅크 코어뱅킹은 **Redis Global Lock + JPA @Lock 이중 보호**로 계좌 잔액 정합성 보장 (SLASH 23)

컬리 실무기(Redisson `tryLock(waitTime, leaseTime)` 설계, 락-트랜잭션 순서): https://helloworld.kurly.com/blog/distributed-redisson-lock/

## 3. 대용량 트래픽 대비

### HikariCP 커넥션 풀
- 우아한형제들 실장애 사례: 한 스레드가 트랜잭션 내에서 커넥션 2개+ 필요할 때 풀 고갈 → 데드락
- **데드락 회피 공식: `pool size ≥ Tn × (Cm − 1) + 1`** (Tn=최대 스레드 수, Cm=스레드당 동시 필요 커넥션 수)
- 출처: https://techblog.woowahan.com/2664/ (이론), https://techblog.woowahan.com/2663/ (실전)

### 인덱스 / 쿼리
- **커버링 인덱스**: 인덱스만으로 조회 완결 → `EXPLAIN`의 Extra가 `Using Where` → `Using Index`로 바뀌는지 확인. DB 최적화의 본질은 **디스크 I/O 감소**
- 페이징: offset 방식은 뒤로 갈수록 느려짐 → **no-offset(커서 기반)** — 카카오페이 정산 배치도 같은 이유로 ZeroOffsetItemReader 자체 제작
- 포트폴리오: 결제 내역 조회 쿼리의 EXPLAIN 전후 비교 리포트

### 캐싱 — 토스 "캐시 문제 해결 가이드" (https://toss.tech/article/cache-traffic-tip)
| 장애 유형 | 문제 | 해법 |
|---|---|---|
| **캐시 쇄도 (Stampede)** | TTL 동시 만료 → DB 부하 폭증 | TTL 지터, 분산락으로 1개 스레드만 재계산, **확률적 조기 재계산(PER)** |
| **캐시 관통 (Penetration)** | 없는 데이터 반복 조회 | 널 오브젝트 캐싱 |
| **캐시 시스템 장애** | Redis 다운 시 DB가 전체 트래픽 감당 불가 | 핵심 기능만 동작하는 failover 설계 |
| **핫키 만료** | 인기 키 만료 순간 폭주 | 분산락으로 단일 쓰기 |

- 카카오페이: Redis 캐시 자체가 병목이 되자 **불변 데이터는 로컬 캐시(2-tier)** 로 전환

### 대기열 (선착순 결제 — 토스페이먼츠 공고 문구 그 자체)
- **Redis Sorted Set**: score=진입 시각, O(log N) 순위 조회
- 일정 배치 크기만큼만 작업 큐로 이동시켜 **DB 유입량 제어**
- 대기 순번은 폴링/WebSocket으로 안내
- 참고 구현: https://github.com/seunggulee1007/flow (티켓팅 대기열)

### 아키텍처 레벨 (배민 주문시스템 — 우아콘 2023, 일 300만 건)
- CQRS: 조회는 역정규화 모델 분리 (배민은 MongoDB)
- 쓰기 한계: 애플리케이션 레벨 샤딩 (주문번호 해시 기반, AbstractRoutingDataSource + AOP)
- 이벤트 아키텍처 + Transactional Outbox로 유실 방지·재발행(replay)
- 영상: https://www.youtube.com/watch?v=704qQs6KoUk

## 4. 부하테스트 — k6 실행 전략

### 도구 선택
- **k6** (권장): JS 시나리오, thresholds로 p95/p99 기준 설정 → 위반 시 실패 처리, CI 통합 용이
- nGrinder: 네이버 오픈소스, 국내 기업 사용 많음 / Gatling: 리포트 우수

### 핵심 지표 읽는 법
- **p99 레이턴시**: p50 10ms / p99 3s면 사용자 1%는 300배 대기
- **p95는 낮은데 p99만 높다** = tail latency 문제 = 다운스트림(DB, 외부 API)의 간헐적 스톨 신호
- GET과 POST는 특성이 완전히 다름 (예: GET 5000 req/s vs POST 200 req/s) → 시나리오 분리
- TPS가 오르다 꺾이는 지점(임계점)에서 에러율·CPU 확인 → 그 지점이 "현재 한계"

### 실무 벤치마크 사례 (월급쟁이부자들 k6 도입기)
- 1,000명 동시 결제 중 50%만 정상 처리 → Slow Query 최적화 → 2,000명 안정 처리
- **이후 실제 프로모션에서 시나리오에 없던 마이페이지 쿼리로 장애** → 시나리오 보강
- 교훈: 부하 시나리오는 결제 API만이 아니라 **실제 사용자 흐름 전체**를 포함해야 함
- 출처: https://medium.com/weolbu/월급쟁이부자들의-부하테스트를-위한-k6-도입기-d7c82e7fe65f

## 5. 관측성 (Observability)

- **메트릭**: Spring Boot Actuator + Micrometer → Prometheus → Grafana
  - 결제 시스템 핵심 패널: **p99 레이턴시, HikariCP 풀 사용률, 서킷브레이커 상태, Kafka consumer lag, 결제 성공률**
- **분산 트레이싱**: Micrometer Tracing + Zipkin/Tempo. 결제→PG→정산 흐름을 traceId로 엔드투엔드 추적
- **구조화 로깅**: JSON 로그 + traceId 포함 (MDC) → 로그-트레이스 연결
- **알림**: actionable한 것만, 심각도 구분
- **SLO 관점** (velog @anlee 사례의 어필 포인트): CPU 같은 인프라 지표가 아니라 **"결제 성공률 / 응답시간" 같은 비즈니스 SLO 기반 모니터링**
- 카카오페이도 APM 계측으로 미추적 구간을 찾아 병목을 발견 — 관측성이 성능 개선의 선행 조건
