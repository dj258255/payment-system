## 2. 채용자 관점 포트폴리오 원칙

- 제로베이스 인사이트: "채용담당자가 보고 싶은 것은 **DB를 설계했다는 사실이 아니라 설계 과정에서 마주한 기술적 문제를 어떻게 해결했는지**" / "클론코딩은 면접관이 쉽게 눈치챈다"
- 신입 포트폴리오 정석: **트러블슈팅 기재는 필수. 문제 → 과정 → 해결 → 결과 순**
- 개발바닥 호돌맨: "기술 하나하나 강의는 많지만 **그것을 합쳐 하나의 서비스를 만드는 과정**을 보여주는 사례가 없다" — 개별 기술이 아니라 시스템으로 통합하는 능력
- 토스페이먼츠 공식 이력서 가이드: "개발 문제 상황을 개선하거나 해결해 본 사항을 **최대한 구체적으로**"
- 핀테크 공통: "금융은 장애보다 **정합성 깨짐**(중복 출금·누락 입금·원장 불일치)이 치명적 — 트랜잭션 경계 + 멱등성 + 재처리 + 보상 + 감사추적을 **한 세트**로 답해야 한다"

### 문서화 3종 세트 (레포에 반드시)
1. **ADR** (`docs/adr/`): "왜 이 기술을 선택했는지"를 마크다운으로 버전관리. 예: "재고 차감에 비관적 락을 선택한 이유 (부하테스트 근거 첨부)"
2. **트러블슈팅 로그**: 문제→과정→해결→결과 템플릿
3. **실험 리포트** (`docs/experiments/`, `docs/performance/`): 장애 주입 실험, 부하테스트 전후 수치·그래프

## 3. 면접 예상 질문 + 답변 골격

### Q1. "사용자가 결제 버튼을 연타(따닥)하면 어떻게 막나요?"
> 1차 프론트 비활성화(UX일 뿐 방어 아님) → 2차 멱등키(Redis 선체크) → **최후 방어선은 DB UNIQUE 제약**. 멱등키 스펙은 토스페이먼츠 방식(키+URL+메서드 조합 판별, 첫 응답 재반환, 처리중이면 409)을 미러링해 구현했다 — 까지 말하면 끝.

### Q2. "환불(승인) API가 타임아웃 났는데 사용자가 다시 누르면? 중복 환불은?" (금융사 실제 면접 질문)
> 타임아웃은 실패가 아니라 **Unknown**(카카오페이 3-상태 모델). ① 같은 멱등키로 재시도하면 PG가 중복 실행하지 않음 ② 조회 API로 실상태 확인 ③ 확인 불가 시 Unknown 저장 → 복구 배치 → 대사가 최종 방어선.

### Q3. "PG 승인은 성공했는데 DB 저장이 실패하면(트랜잭션 롤백) 어떻게 되나요?"
> 돈은 나갔는데 주문이 없는 상태. **보상 트랜잭션(망취소)**을 호출하되, 같은 트랜잭션 안이면 취소 기록도 롤백되므로 **`REQUIRES_NEW` 독립 트랜잭션**으로. 보상 자체가 실패할 수 있으니 보상 요청을 이벤트 테이블에 먼저 남기고 배치가 재시도.

### Q4. "주문과 결제 트랜잭션을 왜 분리하나요?"
> ① 외부 PG 호출은 우리 DB 트랜잭션에 참여할 수 없다(롤백 불가) ② 긴 트랜잭션 = 락 장시간 점유 → 데드락(올리브영 실사례)·커넥션 고갈 ③ 상태머신 + 보상으로 정합성을 맞추는 것이 Saga의 로컬 트랜잭션 분리와 동일한 원리.

### Q5. "동시에 재고 1개를 100명이 결제하면?"
> 락 3종 비교(낙관/비관/분산락)와 **선택 기준**을 말하고, "직접 부하테스트로 비교해 TPS/에러율 근거로 선택했다"고 수치 제시. 선착순 이벤트 규모면 Redis Sorted Set 대기열로 DB 유입 자체를 제어.

### Q6. "결제 완료 이벤트를 Kafka로 쏘는데, DB 커밋 후 발행이 실패하면?"
> Dual write 문제 → **Transactional Outbox**. outbox 테이블에 같은 트랜잭션으로 저장, 릴레이가 발행, 미발행 감지 배치(우아한형제들 방식). at-least-once이므로 컨슈머는 **멱등 컨슈머**(event ID UNIQUE).

### Q7. "웹훅이 두 번 오거나 순서가 바뀌면?"
> 중복: event ID 멱등 처리. 순서: 페이로드를 믿지 않고 **조회 API로 최신 상태를 다시 읽어** 전이(Stripe 공식 권장 + Zero-Payload 발상). 유실: 웹훅+폴링 이중화.

### Q8. "내부 기록과 PG 정산 파일이 안 맞으면?"
> 승인→매입→정산의 시차 때문에 불일치는 구조적으로 발생. **대사 배치**가 transactionKey로 매칭, 4분류(일치/내부만/외부만/금액 불일치), 예외 큐→재대사→티켓. 원장이 복식부기라 **차대변 합계 불일치로 오류가 수학적으로 드러난다**.

### Q9. "PG사 전체 장애가 나면?"
> 서킷브레이커(slowCall 포함)로 빠른 차단 → 스레드 고갈 방지(벌크헤드) → 폴백: 대기 안내 또는 멀티 PG 라우팅. 실제 국내 PG 1시간 장애 사례가 있고, 포트원은 10초 내 트래픽 전환을 세일즈 포인트로 삼는다.

### Q10. "이 시스템 TPS가 얼마나 나오나요? 병목이 뭐였나요?"
> 목표 산정 근거 → 측정 → 병목(구체적 쿼리/구간) → 개선 → 전후 수치. (05 문서의 카카오페이 프레임 그대로. **수치 없이 답하면 감점**)
