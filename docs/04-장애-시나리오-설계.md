# 04. 장애 시나리오 설계 — 외부 API(PG)가 실패할 때

> 외부 PG는 실제로 장애가 난다. 이 문서는 그때 시스템이 어떻게 견디는지의 설계다.
> 신입에게 실무 장애 경험은 없다 → **장애를 직접 재현하고, 대응 설계를 증명하는 것**이 대체 전략.
> 각 시나리오는 실 서비스에서 실제로 발생하는 상황이다.

## 0. 대원칙: 외부 API 호출 결과는 3가지다 (카카오페이)

카카오페이 "MSA 환경에서 네트워크 예외를 잘 다루는 방법"의 핵심 — 결제 도메인 타임아웃 처리의 교과서:

> 외부(PG/제휴사) API 호출 결과를 **성공 / 실패 / Unknown(알 수 없음)** 3가지로 분류하라.
> **타임아웃은 "실패"가 아니라 "Unknown"이다** — 상대 서버에서 처리됐을 수도 있다.

- Unknown 시: 후처리(조회/취소 시도) → 그래도 확인 불가면 **'Unknown 상태'로 저장**하고 사용자에게 재시도 안내 → 이후 배치/대사로 정리
- **읽기 요청은 재시도 안전, 쓰기 요청은 멱등키 있을 때만 재시도**

출처: https://tech.kakaopay.com/post/msa-transaction/

## 시나리오 1. PG 승인 API 타임아웃 — "돈은 빠졌는데 우리는 모른다"

**상황**: 승인 API 호출 → 10초 대기 → 타임아웃. PG에서 승인됐는지 안 됐는지 모름.

**잘못된 처리**: 타임아웃 = 실패로 간주하고 주문 실패 처리 → 고객 카드에서 돈만 빠져나감 (최악의 CS)

**올바른 설계**:
```
1. 결제 상태를 UNKNOWN(승인 미확정)으로 저장
2. 결제 조회 API(GET /v1/payments/{paymentKey})로 실제 상태 확인
   - 승인됨 → 내부 주문 진행 (정상 복구)
   - 승인 안 됨 → 실패 처리
   - 조회도 실패 → 3번으로
3. 망취소 예약: 약 1분 지연 후 취소 API 호출
   (즉시 취소하면 PG측에 결제정보가 아직 없어 실패할 수 있음 — 나이스페이 매뉴얼)
4. 망취소도 실패 → UNKNOWN 상태로 두고 복구 배치 + 대사가 최후 방어선
```

## 시나리오 2. PG 승인 성공 후 내부 처리 실패 — 보상 트랜잭션

**상황**: PG 승인 OK → 재고 차감/주문 저장 중 예외 → 트랜잭션 롤백. **돈은 나갔는데 주문은 없다.**

**함정 (velog @hyezuu 실사례)**: 취소 로직을 같은 트랜잭션 안에서 호출하면 **취소 기록도 함께 롤백**된다.

**올바른 설계**:
```java
// 취소(보상 트랜잭션)는 반드시 독립 트랜잭션으로
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void compensate(String paymentKey, String reason) {
    // 취소 API 호출 + 취소 기록 저장
    // reason에 "망취소" 구분 기재 (토스페이먼츠 공식 안내)
}
```
- 보상 실패 대비: 보상 요청을 **이벤트 테이블에 먼저 기록**(같은 트랜잭션) → 배치가 미처리 보상 건을 재시도
- 이것이 Saga의 보상 트랜잭션을 결제에 적용한 형태 (03 문서)

## 시나리오 3. PG 자체 장애 — 서킷브레이커

**상황**: PG API 서버 장애 (실제로 국내 상위 PG사에서 약 1시간 장애 사례 존재). 모든 요청이 10초씩 대기 → 우리 서버 스레드 고갈 → **PG 장애가 우리 전체 서비스 장애로 전파**.

**설계 (Resilience4j CircuitBreaker)**:
- 상태머신: `CLOSED` → (실패율 임계치 초과) `OPEN` → (대기 후) `HALF_OPEN` → 성공 시 `CLOSED`
- 핵심 설정:
  - `failureRateThreshold` — 실패율 임계치
  - **`slowCallRateThreshold`** — 느린 호출도 실패로 간주 (PG 응답 지연 감지에 중요!)
  - `waitDurationInOpenState`, sliding window
- OPEN 상태에서 예외 대신 **Fallback**: "잠시 후 재시도해 주세요" 안내, 또는 **멀티 PG 라우팅**(아래)
- 함께 쓰는 모듈: **Bulkhead**(PG 호출 전용 스레드풀 격리 — PG가 느려도 다른 API는 정상), **TimeLimiter**(최대 실행시간)

**멀티 PG 폴백**:
- PG 인터페이스를 추상화(어댑터 패턴) → 서킷 OPEN 시 트래픽을 타 PG로 라우팅
- 포트원은 "자체 구축 시 평균 1시간 걸리던 장애 대응을 10초 내 트래픽 전환"으로 홍보
- 부가 효과: 수수료 낮은 PG에 가중 라우팅 (비용 최적화)
- 출처: https://blog.portone.io/opi_multi-pg/

## 시나리오 4. 재시도 폭풍 — 지수 백오프 + 지터

**상황**: PG가 잠깐 흔들릴 때 모든 서버가 동시에 고정 간격 재시도 → **thundering herd / Retry Storm** → PG를 우리가 더 죽임.

**설계**:
- **지수 백오프**: 1 → 2 → 4 → 8분… (토스증권 SLASH 22 실제 적용 간격)
- **+ 지터(jitter)**: 무작위 지연 추가로 재시도 시점 분산
- **재시도 횟수 제한** 필수
- **재시도는 멱등한 요청에만**: 멱등키가 붙은 승인/취소는 재시도 OK, 없는 쓰기는 금지
- **재시도 가능한 에러와 불가능한 에러 구분**:
  - 타임아웃, 5xx, 네트워크 오류 → 재시도 대상
  - 잔액 부족, 한도 초과, 유효하지 않은 카드(4xx 비즈니스 거절) → **재시도해도 소용없음, 즉시 실패 처리**
- 서킷브레이커와 결합: 재시도가 계속 실패하면 서킷이 열려 재시도 자체를 차단 (화해 기술블로그 실무기)

## 시나리오 5. 웹훅이 안 오거나 / 늦게 오거나 / 두 번 오거나 / 순서가 뒤집히거나

| 문제 | 대응 |
|---|---|
| **유실** (우리 서버 다운 중 발송) | 토스는 최대 7회/약 3일 19시간 재전송하지만, 그래도 **폴링 배치**(미확정 결제 주기 조회)를 최후 방어선으로 |
| **중복** (재시도 때문에 정상 동작임) | event ID 기반 `processed_events` UNIQUE 테이블로 멱등 처리 |
| **순서 역전** | 웹훅 페이로드를 믿지 않고 **조회 API로 최신 상태를 다시 읽어서** 상태머신 전이 (Zero-Payload 발상) |
| **처리 지연** | 10초 내 200 응답 필수 → 즉시 200 반환 + 비동기 워커 처리 |

## 시나리오 6. 서버가 죽었다 살아났다 — 미확정 결제 복구 배치

**상황**: 승인 API 호출 직후 우리 서버가 죽음. 재기동 후 `IN_PROGRESS`/`UNKNOWN` 상태로 방치된 결제들.

**설계 — 복구 배치 (인프런 "견고한 결제 시스템 구축" 강의의 '결제 복구' 챕터 주제이기도 함)**:
```
매 N분마다:
1. UNKNOWN / IN_PROGRESS 상태로 T분 이상 머문 결제 스캔
2. 각 건에 대해 PG 조회 API로 실상태 확인
3. 승인됨 → 후속 처리 재개 / 승인 안 됨 → 만료 처리 / 판단 불가 → 망취소
4. 복구 시도 횟수 기록, 초과 시 수동 처리 큐로 + 알림
```
- 복구 배치 자체도 멱등해야 함 (여러 인스턴스가 동시에 돌아도 안전 — 분산락 또는 조건부 UPDATE)

## 시나리오 7. Kafka 컨슈머 처리 실패 — DLQ

**상황**: 결제 완료 이벤트를 소비하던 컨슈머가 계속 실패 (포이즌 메시지) → 파티션 전체가 막힘.

**설계**:
- Spring Kafka `DefaultErrorHandler` + `DeadLetterPublishingRecoverer`: 재시도 N회 후 `<topic>.DLT`로 발행
- DLQ 메시지는 모니터링/알림 → 수동·배치 재처리
- 고급: 재시도 토픽 분리 패턴 (retry-1m / retry-5m / retry-30m 단계별)
- 출처: https://www.daddyprogrammer.org/post/15087/kafka-consumer-retry-deadletter/

## 종합: 다층 방어 (Defense in Depth)

```
1차: 멱등키 + 유니크 제약        → 중복 자체를 차단
2차: 3-상태 모델 + 망취소        → 타임아웃을 안전하게 수습
3차: 서킷브레이커 + 벌크헤드     → 장애 전파 차단
4차: 재시도(백오프+지터) + DLQ   → 일시 장애 자동 복구
5차: 복구 배치 + 웹훅/폴링 이중화 → 남은 미확정 건 정리
6차: 대사 배치                   → 그래도 새는 건 검출 (최종 방어선)
```

> 시스템 설계 책(대규모 시스템 설계 기초 2)의 표현: **"조정(Reconciliation)은 결제 시스템의 최종 방어선"**

## 장애 재현 실험

각 시나리오를 실제로 재현하고 문서화하면 "장애 해결 경험" 요건에 대응된다:

1. **Toxiproxy / WireMock**으로 PG Mock 서버에 지연·오류·타임아웃 주입
2. 시나리오별 실험: "승인 API가 15초 지연되면 우리 시스템은 어떻게 되는가?"
3. Before: 서킷브레이커 없을 때 스레드 고갈 재현 (부하테스트로 수치화)
4. After: 서킷 적용 후 폴백 동작 + 응답시간 유지 확인
5. 각 실험을 `docs/experiments/`에 리포트로 남김 (그래프 포함)
