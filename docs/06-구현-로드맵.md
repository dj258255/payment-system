# 06. 구현 로드맵 — Phase 0~6

> 원칙: **각 Phase가 끝날 때마다 "돌아가는 시스템 + 문서(ADR/실험 리포트) + 블로그 글감"이 하나씩 나온다.**
> 뒤 Phase로 갈수록 차별화 강도가 올라간다. 시간이 부족하면 Phase 4까지만 해도 상위권.
>
> 예상 총 기간: 10~14주 (주 20시간 기준). Phase별 기간은 조정 가능.

## Phase 0. 준비 — 프로젝트 뼈대와 문서화 습관 (0.5주)

**목표**: 처음부터 "실무처럼" 시작하는 기반.

- [ ] Java 21 + Spring Boot 3.x + Gradle 멀티모듈(선택) 스캐폴딩
- [ ] MySQL + Redis를 Docker Compose로 구성, **Testcontainers로 통합 테스트 기반** 마련
- [ ] 토스페이먼츠 개발자센터 가입 → **테스트 API 키 발급** (무료, 샌드박스 제공)
- [ ] `docs/adr/` 디렉토리 생성 — 첫 ADR: "왜 토스페이먼츠인가, 왜 Spring인가" (공고 분석 근거 인용)
- [ ] GitHub 레포: README에 아키텍처 다이어그램 자리, `docs/` 구조 잡기
- [ ] CI (GitHub Actions): 빌드 + 테스트 자동화

**산출물**: 실행 가능한 빈 프로젝트 + ADR-001

> 도메인 선택: "이커머스 주문-결제"를 권장. 교보문고(주문/결제), 무신사, 배민 등 커머스 결제팀 공고와 직결되고, 재고·포인트 같은 동시성 소재가 자연스럽게 나온다.

## Phase 1. 결제 코어 — PG 연동 E2E (2주)

**목표**: 요청 → 인증 → 승인 → 취소 전체 플로우. 여기까지는 "흔한 프로젝트"지만 기초 공사의 품질이 다르다.

- [ ] 주문 도메인: `Order`, `Payment` 분리 설계 (JPA **도메인모델 방식** — 다날/배민 공고 문구)
- [ ] 토스페이먼츠 결제위젯 연동 (간단한 프론트 1페이지면 충분 — 백엔드가 주인공)
- [ ] **승인 API 서버 호출** + **금액 위변조 검증** (요청 전 저장한 amount와 비교 — 토이프로젝트 90%가 빼먹는 부분)
- [ ] **결제 상태머신 구현**: enum + 허용 전이 테이블, 불법 전이 시 예외. 조건부 UPDATE(`WHERE status = :expected`)로 전이
- [ ] 상태 전이 이력 테이블 (append-only `payment_history`)
- [ ] 전액취소 / **부분취소** (cancels 누적, transactionKey 기록)
- [ ] 단위 테스트: 상태머신 전이 전수 테스트 / 통합 테스트: PG Mock(WireMock) 기반

**산출물**: 결제 E2E + 상태머신 테스트 / **블로그 글감**: "결제 상태를 enum 하나로 관리하면 안 되는 이유"

## Phase 2. 실패 설계 — 멱등성과 장애 시나리오 (2~3주) ⭐ 차별화 시작

**목표**: 04 문서의 시나리오 1~6을 전부 구현. "장애 복구 시나리오 구축"(카카오페이 필수 요건) 대응.

- [ ] **멱등키 구현** (토스페이먼츠 스펙 미러링): 멱등키 테이블 + UNIQUE 제약, 409/422 시맨틱, 첫 응답 재반환
- [ ] Redis 선체크 + DB UNIQUE 2중 방어 ("따닥" 방지)
- [ ] **PG 호출 3-상태 모델**: 성공/실패/UNKNOWN 분류, 타임아웃 = UNKNOWN
- [ ] **망취소**: UNKNOWN 건 1분 지연 후 취소 (이벤트 테이블 + 배치 폴링 방식)
- [ ] **보상 트랜잭션**: PG 승인 후 내부 실패 시 `REQUIRES_NEW` 독립 트랜잭션으로 취소
- [ ] **재시도 전략**: 지수 백오프 + 지터, 재시도 가능/불가 에러 구분 (Resilience4j Retry)
- [ ] **서킷브레이커**: Resilience4j, `slowCallRateThreshold` 포함 설정, Fallback 응답
- [ ] **미확정 결제 복구 배치**: UNKNOWN/IN_PROGRESS 방치 건 스캔 → 조회 API → 확정/망취소
- [ ] **장애 재현 실험**: Toxiproxy로 PG Mock에 지연/오류 주입 → 시나리오별 실험 리포트 (`docs/experiments/`)

**산출물**: 장애 시나리오 6종 대응 + 실험 리포트 / **블로그 글감**: "PG 타임아웃은 실패가 아니다 — UNKNOWN 상태 설계기"

## Phase 3. 웹훅 + 이벤트 아키텍처 (2주)

**목표**: 정합성을 이벤트 레벨에서 보장. Kafka 스택 요건(토스페이먼츠·카카오페이) 대응.

- [ ] 웹훅 수신 엔드포인트: **멱등 처리(event ID UNIQUE) → 원본 저장 → 즉시 200 → 비동기 처리**
- [ ] 웹훅 페이로드를 믿지 않고 **조회 API로 재검증** 후 상태 전이 (순서 역전 대응)
- [ ] **웹훅 서명 검증 구현** (Stripe 방식 HMAC-SHA256 + timestamp tolerance — 자체 Mock PG에 적용해서라도 구현. 희소성 높음)
- [ ] 폴링 배치: 웹훅 유실 대비 미확정 결제 주기 조회 (웹훅+폴링 이중화)
- [ ] **Transactional Outbox**: 결제 완료 이벤트를 outbox 테이블에 같은 트랜잭션으로 저장 → Polling Publisher가 Kafka 발행
- [ ] 미발행 이벤트 감지 재발행 배치 (우아한형제들 방식: 5분 이상 미발행 건)
- [ ] **멱등 컨슈머**: processed_event UNIQUE + 동일 트랜잭션
- [ ] **DLQ**: 재시도 N회 후 `.DLT` 토픽, 재처리 어드민 API
- [ ] 컨슈머 예시: 결제 완료 → 알림 발송 / 포인트 적립 (별도 모듈로 분리하면 MSA 흉내)

**산출물**: 이벤트 유실/중복/순서역전 전부 대응 / **블로그 글감**: "웹훅은 믿는 게 아니라 검증하는 것"

## Phase 4. 정산 + 대사 + 원장 (2~3주) ⭐⭐ 최대 차별화

**목표**: "오차 없이 계산하는 정산 시스템"(토스페이먼츠), "대사 시스템 개발"(크림). 취준생 포트폴리오에서 가장 희소한 영역.

- [ ] **복식부기 원장**: transaction—entry 모델, `sum(DEBIT)=sum(CREDIT)` 불변식, append-only 강제
- [ ] 결제/취소/부분취소마다 분개 생성 (사용자 잔액, PG 미수금, 수수료 계정)
- [ ] **정산 배치 (Spring Batch)**: 일 단위 거래 집계 → 결제수단별 수수료 계산 → 가맹점 지급금 생성
  - 집계는 DB GROUP BY/SUM, 기간은 반개구간, 주문 스냅샷 고정 (배민 정산 방식)
  - 대량 데이터(더미 수십만 건) 생성 후 청크/페이징 처리 성능 측정
- [ ] **대사 배치**: PG 거래내역 파일(Mock 생성) vs 내부 기록 매칭
  - 매칭 키 = transactionKey, 결과 4분류(일치/내부만/외부만/금액 불일치)
  - 불일치 → 예외 큐 → 자동 재대사 → 초과 시 수기 티켓
  - **일부러 불일치 데이터를 심어서** 검출되는 것을 테스트로 증명
- [ ] Clearing 검증: 클리어링 계정 잔액 0 수렴 체크 (Stripe Ledger 방식)

**산출물**: 원장+정산+대사 전체 파이프라인 / **블로그 글감**: "복식부기로 결제 데이터 정합성을 수학적으로 증명하기"

## Phase 5. 성능 개선 — 수치가 있는 리포트 (2주) ⭐⭐

**목표**: 카카오페이 스토리 프레임 그대로. "수십배 트래픽에도 무너지지 않는 시스템"(토스페이먼츠) 대응.

- [ ] **목표 산정**: "선착순 1만 명 쿠폰 결제 이벤트" 시나리오 → 목표 TPS 정의 (근거 문서화)
- [ ] **k6 부하테스트**: 현재 한계 측정 (TPS, p95/p99, 에러율) — 실제 사용자 흐름 전체 시나리오
- [ ] **병목 계측**: APM(Pinpoint/Grafana Tempo) + EXPLAIN + slow query 로그
- [ ] 개선 사이클 (하나씩 적용 → 재측정 → 기록):
  - 커넥션 풀 튜닝 (데드락 공식 적용)
  - 조회 쿼리 인덱스/커버링 인덱스 (EXPLAIN 전후)
  - 캐싱 (TTL 지터, 널 오브젝트 — 토스 가이드 4유형 반영)
  - **동시성 비교 실험**: 재고 차감을 비관적 락 vs 낙관적 락 vs Redisson으로 각각 구현 → 부하테스트로 TPS/에러율 비교 → 선택 근거 ADR화
  - 선착순 이벤트용 **Redis Sorted Set 대기열**
- [ ] **최종 리포트**: 목표→한계→병목→개선→전후 수치 (그래프 포함, `docs/performance/`)

**산출물**: 부하테스트 리포트 + 락 비교 실험 / **블로그 글감**: "재고 차감 락 3종 비교 — 수치로 고르기"

## Phase 6. 운영 완성도 — 관측성 + 어드민 + 마무리 (1~2주)

**목표**: "운영"의 흔적. 다날(ELK), 크림(모니터링) 요건 대응.

- [ ] **Grafana 대시보드**: 결제 성공률(SLO), p99, 서킷 상태, Kafka lag, HikariCP 사용률
- [ ] 구조화 로깅(JSON + traceId), 분산 트레이싱 (결제→PG→이벤트 흐름)
- [ ] 임계치 알림 (결제 실패율 급증 시)
- [ ] **백오피스 어드민 최소 구성**: 결제 조회/강제 취소/대사 불일치 수기 처리/DLQ 재처리 — "백오피스"가 직무명으로 등장하는 도메인임
- [ ] README 완성: 아키텍처 다이어그램, 장애 시나리오 표, 성능 리포트 요약, ADR 목록 링크
- [ ] (선택) 무중단 배포 데모: Docker + 롤링 배포, 배포 중 결제 요청 유실 없음 증명

**산출물**: 운영 대시보드 + 어드민 + 완성된 README
