# 02. 결제 도메인 핵심 개념 — 한국 결제 생태계 이해

> 결제가 어떻게 흘러가는지 이해하기 위한 도메인 지식.
> 주 출처: 토스페이먼츠 공식 개발자 문서 (docs.tosspayments.com)

## 1. 참여자 구조: 가맹점 → PG → 카드사/VAN

```
구매자 ──> 가맹점(우리 서버) ──> PG사(토스페이먼츠 등) ──> 카드사
                                        │
                                       VAN (승인 전문 통신망)
```

- **PG(Payment Gateway)**: 온라인 가맹점을 대신해 카드사 계약·정산을 대행하고 수수료를 받는 사업자. 토스페이먼츠, KG이니시스, 나이스페이먼츠, 다날 등
- **VAN(Value Added Network)**: 상점과 카드사 사이에서 거래 정보(승인 전문)를 주고받는 통신망 사업자
- **간편결제**(카카오페이, 네이버페이, 토스페이)는 PG 위에 얹힌 지갑/인증 레이어로 이해

## 2. 결제 3단계: 요청 → 인증 → 승인 (토스페이먼츠 공식 정의)

이 3단계 분리가 결제 시스템 설계의 출발점이다.

### ① 요청 (Request)
'결제하기' 클릭 → 토스페이먼츠 SDK의 결제 요청 메서드 호출 → 결제창 오픈.
이때 `orderId`(가맹점 채번), `amount`, `successUrl`/`failUrl` 전달.

### ② 인증 (Authentication)
결제창에서 카드번호·CVC 입력 또는 앱카드/간편결제 → **카드사가 구매자를 카드 소유자로 인증**.
인증 성공 시 `paymentKey` 발급 + `successUrl`로 리다이렉트 (쿼리에 `paymentKey`, `orderId`, `amount`).

### ③ 승인 (Approval)
**가맹점 서버가 시크릿 키로 승인 API(`POST /v1/payments/confirm`) 호출** → PG가 카드사에 최종 승인 요청.

> **왜 요청과 승인을 분리하나?**
> 토스페이먼츠 공식 답변: "데이터 정합성을 보장하고, 상점에서 직접 해야 할 일을 줄이기 위해."
> 한 번에 처리하면 비동기 처리로 인한 데이터 불일치 위험이 커진다. **승인의 최종 트리거를 가맹점 서버가 쥐게 하는 설계** — 서버가 재고 확인, 금액 검증을 마친 뒤에만 돈이 움직인다.

### 금액 위변조 검증 (필수 구현 — 토이프로젝트 대부분이 빼먹는 부분)

`successUrl`의 쿼리 파라미터는 클라이언트를 거쳐 오므로 조작될 수 있다.

```
1. 결제 요청 전: 서버에 (orderId, amount) 저장
2. successUrl 리다이렉트 후: 돌아온 amount와 저장된 amount 비교
3. 일치할 때만 승인 API 호출. 불일치 → 승인 거부 + 알림
```

토스페이먼츠 배포 체크리스트에 "클라이언트 요청 금액과 실제 결제 금액 일치 확인 필수"로 명시되어 있다.

### 결제창 방식 vs 키인(Key-in) 방식
- **결제창(인증 결제)**: 카드사 인증을 구매자가 직접 수행 — 일반적인 방식
- **키인(수기/비인증 결제)**: 카드번호만으로 결제. 제한 업종 + 엄격한 심사 필요. 이 시스템은 결제창(위젯) 방식을 쓴다

## 3. 취소: 전액취소 / 부분취소 / 망취소

### 취소 API
`POST /v1/payments/{paymentKey}/cancel`, `cancelReason` 필수.
- `cancelAmount` 생략 → 전액취소 / 지정 → **부분취소**
- 부분취소를 여러 번 하면 응답의 `cancels` 배열에 취소 객체가 누적, 각 건마다 고유 `transactionKey` 발급
- 가상계좌: 입금 전 부분취소 불가, 입금 후 환불은 환불계좌(`refundReceiveAccount`) 필수

### 망취소 (Network Cancel) — 결제 도메인 이해도를 보여주는 핵심 키워드

**정의**: 승인 요청 후 타임아웃·응답 유실 등 비정상 실패 시 결제를 취소해 원복시키는 처리.
클라이언트는 실패로 판단했지만 PG/카드사에서는 승인이 완료된 **"유령 승인"**을 정리하는 것이 목적.

**토스페이먼츠 정책**: 별도 망취소 API 없음. **일반 취소 API를 쓰되 `cancelReason`에 망취소임을 구분 기재**하라고 안내. (DB 저장 중 에러, 재고 없음 등으로 승인 직후 롤백해야 하는 경우 포함)

**실무 구현 포인트** (taes-k 사례 + 나이스페이 매뉴얼):
- **타임아웃 직후 즉시 취소하면 PG측에 아직 결제정보가 생성되지 않아 망취소가 실패할 수 있다 → 약 1분 지연 후 요청**
- 망취소 유효기간은 통상 1시간 (나이스페이 기준)
- 아키텍처 선택지: ① Kafka 지연 발행 ② 이벤트 테이블 + 1분 주기 배치 폴링 → 실무에선 "확실한 처리"가 중요해 **배치 방식** 선택 (재시도 용이, 구현 단순)
- 망취소 모듈은 일반 결제 프로세스와 물리적으로 분리
- 결제 요청 시 가맹점 채번 요청 ID를 반드시 포함해 추적 가능하게

## 4. 결제 상태머신

### 토스페이먼츠 Payment.status (공식)

```
READY ──> IN_PROGRESS ──> DONE ──> CANCELED / PARTIAL_CANCELED
  │            │            ↕ (가상계좌 입금오류 시 역전이!)
  │            ├──> ABORTED (승인 실패)
  └────────────┴──> EXPIRED (30분 경과)
        WAITING_FOR_DEPOSIT (가상계좌 입금 대기)
```

| 상태 | 의미 |
|---|---|
| `READY` | 결제 생성 초기 상태 |
| `IN_PROGRESS` | 인증 완료 (이 상태에서 승인 API 호출하면 완료) |
| `WAITING_FOR_DEPOSIT` | 가상계좌 입금 대기 |
| `DONE` | 승인 완료 |
| `CANCELED` / `PARTIAL_CANCELED` | (부분) 취소됨 |
| `ABORTED` | 승인 실패 |
| `EXPIRED` | 유효시간 30분 경과로 거래 취소 |

**주의**: 가상계좌에서 송금한도 초과·네트워크 이슈로 입금 오류 시 **DONE → WAITING_FOR_DEPOSIT 역전이**가 발생할 수 있다. 즉, 상태는 웹훅으로 계속 추적해야 한다.

### Stripe PaymentIntent (글로벌 비교 대상)

`requires_payment_method` → `requires_confirmation` → (`requires_action`: 3DS) → `processing` → `succeeded` / `canceled`. 실패 시 `requires_payment_method`로 회귀. `succeeded`, `canceled`만 최종(terminal) 상태.

### 우리 서버의 상태머신 설계 원칙

1. **PG 상태와 별개로 내부 주문/결제 상태머신을 둔다**
2. **허용된 전이만 코드로 강제** — enum + 전이 테이블, `CANCELED → DONE` 같은 불법 전이는 예외 발생
3. 전이는 **조건부 UPDATE** (`WHERE status = :expected`)로 낙관적 동시성 제어
4. **상태 전이 이력(payment_history)을 append-only로 기록** → 감사·대사에 활용
5. 웹훅/폴링/사용자 액션이 동시에 같은 결제를 전이시킬 수 있으므로 **멱등 전이** (이미 목표 상태면 no-op)

## 5. 정산과 대사 (개념 — 구현은 03 문서)

- **정산(Settlement)**: 구매자 결제금액에서 PG/플랫폼 수수료를 제외한 금액을 가맹점(판매자)에게 지급하는 프로세스. 결제수단별 수수료율이 다름
- **대사(Reconciliation)**: 내부 기록(주문/결제/원장)과 외부 기록(PG 정산 파일, 카드사 매입 내역)을 건 단위로 대조해 불일치를 찾는 작업
- 신용카드는 **승인 → 매입(전표 제출) → 정산 입금**의 단계적 처리라 각 단계 간 시차·불일치가 구조적으로 발생 — 이게 대사가 필요한 근본 이유
