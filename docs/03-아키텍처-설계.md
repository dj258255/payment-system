# 03. 아키텍처 설계 — 멱등성 · 분산 트랜잭션 · 원장 · 웹훅 · 정산/대사

> 결제 시스템의 뼈대가 되는 5개 설계 패턴. 각각이 면접 질문 하나씩에 대응한다.

## 1. 멱등성 (Idempotency) — "따닥"과 안전 재시도의 해법

### 개념
- GET/PUT/DELETE는 본질적으로 멱등, **POST는 멱등하지 않으므로 결제 승인·취소 같은 POST API에 멱등키가 필요**
- Stripe: "charge API를 실수로 두 번 호출하면 고객이 이중 청구되는데, 이는 매우 나쁘다(double-charged, which is very bad)"
- 멱등성이 있으면 "실제 결제됐는지 확인하지 않고도 **안심하고 재시도**"할 수 있다 — 타임아웃 처리의 전제 조건

### 토스페이먼츠 공식 구현 (그대로 따라 만들면 됨)
- `Idempotency-Key` HTTP 헤더, 최대 300자, UUID 권장, **유효기간 15일**
- 중복 판별: **API 키 + API 주소 + HTTP 메서드 + 멱등키** 조합
- 서버 동작: ① 헤더에서 멱등키 확인 → ② 이전 처리 기록 조회 → ③ 있으면 **저장된 첫 응답을 그대로 반환(재처리 없음)** / 없으면 처리 후 응답 저장
- 에러 시맨틱 (우리 서버에도 동일하게 구현할 것):
  - `409 IDEMPOTENT_REQUEST_PROCESSING` — 이전 요청이 아직 처리 중인데 재요청
  - `422` — 같은 멱등키로 **다른 요청 본문** 전송
  - **에러 난 요청을 멱등키를 바꿔 재시도하는 것은 위험** (중복 실행 가능성) — 공식 문서 명시

### 우리 서버 구현 설계
```
멱등키 테이블: (idempotency_key UNIQUE, request_hash, response_body, status[PROCESSING/DONE], created_at)

1. 요청 수신 → 멱등키로 INSERT 시도 (UNIQUE 제약)
2. INSERT 성공 → 신규 요청, 비즈니스 처리 → 응답 저장
3. INSERT 실패(중복) →
   - status=DONE → 저장된 응답 반환
   - status=PROCESSING → 409 반환 (동시 요청)
   - request_hash 다름 → 422 반환
```
- 2중 방어: Redis 멱등키 선체크(빠른 차단) + **DB UNIQUE 제약(최후 방어선)**
- `orderId`에도 UNIQUE 제약 — 가맹점 서버 자체의 멱등성
- 승인 응답 저장과 주문 상태 변경은 **한 트랜잭션**으로

### Stripe 클라이언트 3원칙 (우리가 PG를 호출할 때)
① **일관성**: 모든 원격 호출 실패는 재시도로 처리 ② **안전성**: 멱등키로 무한 재시도 가능하게 ③ **책임감**: exponential backoff(2^n) + jitter로 thundering herd 방지

## 2. 분산 트랜잭션 — Saga & Transactional Outbox

### 왜 2PC를 안 쓰나 (면접 단골)
microservices.io에 "2PC is not an option"으로 명시. 이유:
- 참여자 전부 살아있어야 커밋 → **가용성 저하**
- 시스템 간 강결합, NoSQL/메시지 브로커의 XA 미지원
- **결정적 이유: PG(외부 시스템)는 우리 트랜잭션에 참여할 수 없다** — 외부 API 호출은 롤백이 안 되므로 애초에 로컬 트랜잭션 + 보상으로 설계해야 함

### Saga 패턴
- 정의: "로컬 트랜잭션의 연속. 각 로컬 트랜잭션은 DB를 갱신하고 다음을 트리거하는 이벤트를 발행"
- **Choreography**(이벤트로 분산 의사결정) vs **Orchestration**(중앙 오케스트레이터가 명령)
  - 결제처럼 **흐름 추적·보상 순서가 중요한 도메인은 Orchestration이 관리 용이**
- **보상 트랜잭션**: 중간 실패 시 이전 단계들을 역순으로 되돌리는 트랜잭션을 명시적으로 설계. 결제에서 **망취소가 대표적인 보상 트랜잭션**
- 예: 주문 생성 → PG 승인 → 포인트 차감 → 재고 확정. 포인트 차감 실패 시 → PG 취소(망취소) → 주문 실패 처리
- **격리성(I) 부재 문제**: 여러 saga의 동시 실행이 데이터 이상을 유발할 수 있음 → semantic lock(상태 필드로 '처리중' 표시) 등 대응 필요

### Transactional Outbox — "DB와 Kafka에 동시에 쓰지 마라"
- **해결하는 문제(dual write)**: DB 커밋 후 Kafka 발행 실패 → 이벤트 유실 / 반대 순서 → 유령 이벤트
- 동작: 비즈니스 데이터 변경 + **outbox 테이블 INSERT를 같은 로컬 트랜잭션**으로 커밋 → 별도 Message Relay가 outbox → Kafka 발행
- Relay 구현: ① **Polling Publisher**(주기 폴링 — 구현 단순, 포트폴리오 권장) ② CDC/Debezium(binlog 추적)
- 결과: **at-least-once 보장** (트랜잭션 성공 = 이벤트 반드시 발행) → 소비자는 반드시 **멱등 컨슈머**로

### 국내 실사례
- **우아한형제들 회원시스템**: Outbox를 도메인과 동일 RDBMS에 구현. **배치가 5분 이상 미발행 이벤트를 감지해 자동 재발행** + 기간/이벤트타입별 **replay** 가능. 외부 이벤트는 **ZERO-PAYLOAD**(식별자+행위+시각만) — https://techblog.woowahan.com/7835/
- **배민스토어**: 이벤트 **순서 역전** 문제([수정 → 등록] 순서로 수신)를 Zero-Payload로 우회 — 이벤트엔 ID만 담고 소비자가 원본 API로 최신 상태 조회 → 항상 최종 상태만 반영 — https://techblog.woowahan.com/13101/
- **토스뱅크 코어뱅킹**: 필수 트랜잭션만 동기, 세금처리 등은 Kafka 비동기 분리 (DML 80회→50회) — https://toss.tech/article/slash23-corebanking

### 멱등 컨슈머 구현
- Kafka는 기본 at-least-once (중복 가능). exactly-once semantics도 **컨슈머 측 중복 처리는 보장 안 됨**
- 실무 정답: **at-least-once + 멱등 컨슈머** — 이벤트ID 기반 `processed_event` 테이블(UNIQUE) + 비즈니스 처리를 동일 트랜잭션으로
- 순서 민감 데이터는 파티션 키(주문 ID) 고정

## 3. 원장 (Ledger) — 복식부기, append-only

> 공고 표면엔 드물지만 면접에서 도메인 깊이를 증명하는 최고의 카드. Stripe Ledger가 교과서.

### Stripe Ledger 핵심 (stripe.dev 공식)
- 구성: **계정(account)** = "유형과 속성으로 구분되는 자금 버킷" + **이벤트(event)** = 계정 간 자금 이동 기록
- **복식부기 원칙**: 모든 차변(debit)에 대응하는 대변(credit), **차변 총합 = 대변 총합**이 항상 성립 → 자금 이동의 "수학적 정합성 증명"
- **Immutable / Append-only**: 발행된 거래는 삭제·수정 불가. 잔액은 덮어쓰지 않고 **이벤트 두 줄(차/대변)을 추가**. 전체 replay로 임의 시점 상태 재구성 가능 → 감사(audit) 보장
- **정합성 검증 3축**: ① **Clearing** — 중간 계정 잔액이 0으로 수렴해야 함(누락 거래는 잔액 불균형으로 즉시 드러남) ② **Timeliness** — 타임스탬프 임계값 ③ **Completeness** — 원천 DB의 모든 ID가 Ledger 이벤트와 매칭
- 규모: 일 50억 이벤트, 달러 볼륨 99.99%가 4일 내 검증

### 우리 프로젝트 구현 설계
```
transaction (거래) 1 ── N entry (분개)
entry = (account_id, direction[DEBIT/CREDIT], amount)
불변식: 트랜잭션 단위로 sum(DEBIT) = sum(CREDIT)
```
- 계정 예: 사용자 잔액, PG 미수금(receivable), 수수료 수익, 클리어링 계정
- 결제 승인 → [PG미수금 +10,000 / 매출 -10,000] 식으로 분개 생성
- entry 테이블은 **UPDATE/DELETE 금지** (DB 권한 또는 애플리케이션 불변식으로 강제)
- 잔액 = entry 합계. 성능용 스냅샷은 캐시일 뿐, **원천은 항상 저널**
- 음수 잔액 방지: 조건부 UPDATE (`WHERE balance >= :amount`) 또는 락

## 4. 웹훅 처리 — 유실 · 중복 · 순서역전 대응

### 토스페이먼츠 웹훅 스펙
- 주요 이벤트: `PAYMENT_STATUS_CHANGED`, `DEPOSIT_CALLBACK`(가상계좌 입금), `CANCEL_STATUS_CHANGED` 등 10종
- **재시도: 최대 7회**, 간격 1→4→16→64→256→1024→4096분(지수 백오프), **최초 실패 후 약 3일 19시간까지**
- 응답 요건: **10초 이내 HTTP 200**
- **순서 보장: 문서에 명시 없음** = 보장 안 된다고 가정하고 설계
- **서명 검증 메커니즘도 문서에 명시 없음** → 한국 PG 웹훅은 **"신뢰하지 말고 조회 API로 재검증"이 사실상 표준** (포트원도 동일 권장)

### Stripe 웹훅 (글로벌 베스트 프랙티스 — 포트폴리오에서 구현할 가치)
- **서명 검증**: `Stripe-Signature: t=<timestamp>, v1=<sig>` → `signed_payload = timestamp + "." + raw_body`를 endpoint secret으로 **HMAC-SHA256** → **constant-time 비교**(타이밍 공격 방지)
- **Replay attack 방지**: timestamp가 서명에 포함 + **tolerance 5분** 초과 이벤트 거부
- **순서 보장 안 됨 공식 명시** → 이벤트는 알림으로만 쓰고 **필요 데이터는 API로 최신 상태 조회** (Zero-Payload와 같은 발상)
- **중복 이벤트**: 처리한 event ID를 `processed_events`(UNIQUE)에 기록하는 멱등 핸들러 필수
- **빠른 2xx 후 비동기 처리**: 핸들러에서 무거운 로직 돌리면 타임아웃 → 즉시 200 반환하고 큐로

### 수신 파이프라인 종합 설계
```
[서명/출처 검증] → [event ID 멱등 체크] → [원본 페이로드 저장(감사용)]
→ [즉시 200 응답] → [비동기 워커: 조회 API로 실상태 확인 → 상태머신 전이]
```
- 웹훅 유실 대비: 웹훅만 믿지 말고 **주기적 폴링 배치**(미확정 결제 조회) + **대사 배치**를 최후 방어선으로

## 5. 정산(Settlement)과 대사(Reconciliation)

### 정산 배치 설계 (우아한형제들 정산시스템 미러링)
우아한형제들 프로세스: 주문 수집 → 조정 데이터(보상금 등) 반영 → **지급금 생성** → 지급

Spring Batch 구현 포인트:
- Reader(조회) / Processor(가공, 멀티스레드) / Writer(저장) 분리
- 집계는 WAS 메모리 대신 **DB에서 GROUP BY/SUM**
- 기간 경계는 나노초 포함 `start ≤ statusAt < end` **반개구간**
- 주문 상태는 **Snapshot 엔티티**로 시점 고정 (정산 중 주문이 바뀌어도 무관하게)
- "정산은 데이터 정합성과 트랜잭션, 대용량 데이터 처리가 핵심인 시스템" — 배민 정산팀
- 대용량 최적화(카카오페이): Partitioning + Cursor 기반 ItemReader + Bulk Operation (오프셋 페이징은 뒤로 갈수록 느려짐 → ZeroOffsetItemReader)

### 대사 엔진 설계 (Modern Treasury 3단계 아키텍처)
1. **Data Ingestion** — PG 정산 파일 수집 (D+1~D+2 도착)
2. **Data Normalization** — 포맷 파싱·정규화
3. **Data Reconciliation** — 거래 레벨 매칭 (1:1, N:1 — 여러 결제가 하나의 입금으로 묶이는 경우)

핵심 철학: **코어 엔진은 완전 결정적(deterministic)** — 같은 입력이면 100% 같은 결과. 엔진이 99%를 자동 매칭하고 확신 못하는 1%만 사람이 처리. "AI가 자동 확정"이 아니라 "**결정적 매칭 + 예외만 사람**"이 정석.

### 대사 결과 분류와 처리
| 분류 | 의심 원인 | 처리 |
|---|---|---|
| 완전 일치 | — | 통과 |
| 내부에만 존재 | 망취소 실패, 매입 누락 | 예외 큐 → 재대사 → 티켓화 |
| 외부에만 존재 | 웹훅 유실, 내부 저장 실패 | 예외 큐 → 조회 API 보정 |
| 금액 불일치 | 부분취소 시점 차이 | 시점 보정 후 재대사 |

- 매칭 키는 승인일 기준이 아닌 **거래번호(transactionKey/tid)** 사용
- 대사 배치는 **멱등하게** (같은 파일 재실행 시 결과 동일)
- 불일치 건: 자동 재대사(다음 배치 주기) → 기간 초과 시 수기 처리 티켓화
